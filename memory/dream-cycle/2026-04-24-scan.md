### 1. [Anthropic Post-Mortem: Claude Performance Degradation Explained](https://venturebeat.com/technology/mystery-solved-anthropic-reveals-changes-to-claudes-harnesses-and-operating-instructions-likely-caused-degradation)
**Source:** VentureBeat / Anthropic Blog (Apr 23) | **Relevance:** high
**Summary:** Anthropic published a detailed post-mortem identifying 3 product-layer changes that caused widespread perceived degradation: (1) reasoning effort lowered from high→medium in Claude Code (Mar 4), (2) caching bug that wiped short-term memory after inactivity (Mar 26), (3) system prompt verbosity limits that hurt coding quality (Apr 16). Model weights were NOT changed. Fixes shipped in v2.1.116.
**Why it matters for us:** Validates our decision to stay on Opus 4.6 and avoid Opus 4.7. The caching bug could explain intermittent quality drops we saw. API users were unaffected, which means our direct API usage via OpenClaw was fine, but Claude Code sessions may have been hit.

### 2. [Google ReasoningBank: Agent Memory That Learns from Failures](https://research.google/blog/reasoningbank-enabling-agents-to-learn-from-experience/)
**Source:** Google Research / ICLR 2026 | **Relevance:** high
**Summary:** Google open-sourced ReasoningBank, a memory framework that distills reasoning strategies from both successful AND failed agent experiences. Shows 8.3% higher success rate on WebArena and 4.6% on SWE-Bench vs memory-free agents. Introduces Memory-aware Test-time Scaling (MaTTS) for parallel exploration.
**Why it matters for us:** This is exactly what our dream cycle does manually (learn from corrections/failures). ReasoningBank could automate the corrections→MEMORY.md→behavior pipeline. The structured memory format (title/description/content) maps cleanly to our existing memory architecture. Open-source on GitHub.
