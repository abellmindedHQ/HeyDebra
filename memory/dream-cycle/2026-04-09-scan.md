### [Research-Driven Agents: When Agents Read Before They Code](https://blog.skypilot.co/research-driven-agents/)
**Source:** Hacker News #14 (144 points) | **Relevance:** high
**Summary:** SkyPilot team added a literature research phase to the autoresearch/pi-autoresearch loop. Before writing optimizations for llama.cpp, the agent read arXiv papers and studied competing forks (ik_llama.cpp, CUDA backend). Result: 5 optimizations landed (+15% flash attention on x86, +5% ARM) for $29 total. Code-only agents missed these because the answer lived outside the source.
**Why it matters for us:** Our dream-cycle already does a version of this (research scan before proposals). But we could apply this to coding tasks specifically — when Alex asks for a new feature or skill, reading existing skill implementations and relevant docs before touching code would improve quality. The "read forks and competitors first" heuristic is immediately actionable.

### [I Still Prefer MCP Over Skills](https://david.coffee/i-still-prefer-mcp-over-skills/)
**Source:** Hacker News #4 (22 points) | **Relevance:** medium
**Summary:** Author argues MCP (Model Context Protocol) is architecturally superior to Skills for service integrations: remote MCP servers need no local install, handle auth gracefully (OAuth), update automatically, are naturally sandboxed, and support smart tool discovery. Skills are good for pure knowledge/behavior tuning but CLIs don't work in web-based Claude/ChatGPT contexts.
**Why it matters for us:** We're a Skills-heavy setup. This is fair pushback. Our Skills depend on CLIs installed on the Mac mini — they'd break in any non-local context. If Alex moves toward cloud/mobile-first workflows, we should have MCP fallbacks for key integrations. Also: OpenClaw's architecture may be heading toward MCP natively.
