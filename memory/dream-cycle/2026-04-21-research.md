## Deep Dive 2: OpenAI Chronicle vs Our Memory Architecture

### What It Is
Chronicle is OpenAI's answer to the "AI amnesia" problem. It captures screenshots of your screen via background agents, OCRs them, and stores memories as local Markdown files. Opt-in for Pro on macOS. Key claims: 40% productivity boost, cross-session context, "Personal AGI" positioning.

### Architecture Comparison

| Feature | Chronicle | Our System |
|---------|-----------|-----------|
| Memory storage | Local Markdown | Local Markdown (MEMORY.md, daily notes) |
| Capture method | Screenshot OCR | Explicit text capture + tool use logging |
| Retrieval | Searchable (details unknown) | memory_search (Gemini embeddings) + qmd (BM25 + semantic) |
| Cross-session | Yes (persistent store) | Yes (active-context.md + MEMORY.md + daily notes) |
| Self-improvement | Unknown | Dream cycle (nightly research + reflection + proposals) |
| Privacy | Screenshots sent to OpenAI servers for OCR | Everything local, never leaves machine |
| Graph memory | No | Neo4j + SecondBrain wikilinks |
| Structured memory | Flat | Hierarchical (daily → long-term → corrections → active context) |

### Key Insight
**We're architecturally ahead of Chronicle in every dimension except ease of setup.** Chronicle is "install and forget" — it passively captures everything. Our system requires intentional capture but is richer, more private, and more structured.

The one thing Chronicle does that we don't: **passive capture from screen context.** We only capture what explicitly flows through OpenClaw. If Alex is working in a browser or app that doesn't integrate with us, we miss it. VisionClaw (Ray-Ban Meta) was supposed to address this but is stalled.

### Implementation Path
Nothing to implement. This validates our approach. The one takeaway: consider a lightweight passive capture mechanism for things that happen outside OpenClaw sessions (browser history, active app tracking). But this is a VisionClaw / Kapture MCP territory and not urgent.

### Trade-offs
Chronicle's passive approach is convenient but creepy (screenshots, OCR on remote servers). Our intentional approach requires more effort but is more trustworthy.

### Recommendation
**Watch it.** No action needed. Our architecture is better. But study their memory pruning/dashboard UX if they open-source any of it.

## Deep Dive 4: BB Per-Group SystemPrompt (from OC 2026.4.20)

### What It Is
New OpenClaw feature allowing per-group chat system prompt configuration. Instead of one global BB behavior, each group chat can have its own personality/rules.

### How We'd Use It

```json
{
  "plugins": {
    "bluebubbles": {
      "groups": {
        "any;+;26496e31a52b4d7091da7b17a5a1380d": {
          "systemPrompt": "You're in KBUDDS (Alex + Marshall + Everett). Keep it bro. Sports, dad jokes, light roasting. No work talk unless someone brings it up."
        },
        "any;+;63d1dbe0006d46abbc3aa07a4fb38c8b": {
          "systemPrompt": "You're in the Family chat (Alex + Hannah + Sallijo). Warm, helpful, family-focused. Be gentle with Sallijo. Never bring up pregnancy."
        },
        "any;+;a96e1f6eaaba404abd15b7b1a1a1cdea": {
          "systemPrompt": "You're in Alex + Chelsea (therapist) chat. Clinical awareness, emotional intelligence, HeartMath/Mirror context. Professional but warm."
        },
        "*": {
          "systemPrompt": "Default: casual, warm, short messages. Match the room energy."
        }
      }
    }
  }
}
```

### Implementation Path
1. Upgrade to 2026.4.20 (prerequisite)
2. Define systemPrompts for all known group chats
3. Test with low-stakes chats first (Tyler, KBUDDS)
4. Roll out to sensitive chats (Family, Chelsea) after validation

### Recommendation
**Do it** after the upgrade (Thu/Fri). This is the best BB improvement in months.
