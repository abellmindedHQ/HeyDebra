## Deep Dive 2: YourMemory — Biological Decay for Agent Memory

### What It Is
YourMemory is an MCP-based persistent memory layer that applies Ebbinghaus forgetting curves to AI agent memory. Key mechanics:

**Memory Decay Formula:**
```
effective_λ = base_λ × (1 - importance × 0.8)
strength = clamp(importance × e^(−effective_λ × active_days) × (1 + recall_count × 0.2), 0, 1)
```

**Category-based survival times:**
- Strategy memories: ~38 days without recall
- Fact memories: ~24 days
- Assumptions: ~19 days
- Failures: ~11 days

**Key innovations:**
- Hybrid retrieval: BM25 keyword + cosine similarity (0.4/0.6 weight)
- Graph expansion: BFS from initial hits surfaces related-but-different-vocabulary memories
- Chain-aware pruning: decayed memories kept alive if graph neighbors are strong
- Active-days tracking: vacations don't cause memory loss
- Multi-agent support with private/shared memory scoping

**Benchmarks:** 59% Recall@5 on LoCoMo (vs 28% Zep Cloud). 85% Recall-all@5, 87% nDCG@5.

### How It Applies to Us
Our current memory system:
- `MEMORY.md`: manually curated, never decays, grows indefinitely
- `memory/YYYY-MM-DD.md`: daily logs, never pruned
- `memory/corrections.md`: append-only
- `memory/dream-cycle/`: 27 cycles of files, never cleaned
- `qmd`: semantic search over vault, no decay scoring

**Problems YourMemory could solve:**
1. **Stale memory**: Our MEMORY.md has items from March that may no longer be relevant (e.g., Paperclip team details for a shut-down project)
2. **Dream cycle proposals**: 27 cycles of proposals, zero reviewed. Decay would naturally prune unacted proposals
3. **Memory search noise**: qmd returns stale results without recency weighting
4. **Corrections decay**: "Failure" category with ~11 day survival is interesting. Once a correction is promoted to MEMORY.md, the original correction.md entry could decay
5. **Graph expansion**: Could connect related memories across our fragmented file system

**Architectural tension:** YourMemory is MCP-based (Python), our stack is Node.js. It uses DuckDB, we use SQLite (qmd). Integration would require either:
- Running as a separate MCP server alongside OpenClaw
- Extracting the decay algorithm and applying it to our existing file-based system
- Hybrid: use YourMemory for memory operations, keep files as canonical source

### Implementation Path
1. **Phase A (trivial):** Apply decay concepts manually to dream cycle. Tag proposals with importance scores. Auto-archive proposals older than 14 days with no action.
2. **Phase B (moderate):** Build a decay scoring layer on top of qmd. Weight search results by recency × importance × recall frequency.
3. **Phase C (significant):** Integrate YourMemory as MCP server. Migrate canonical memory from files to YourMemory, keep files as backup/human-readable layer.

### Trade-offs
- **Gain**: Natural garbage collection for growing memory, better recall relevance, reduced token waste from stale context
- **Cost**: Phase C is a major migration. DuckDB dependency. MCP coordination overhead.
- **Risk**: Aggressive decay could lose important context. Need careful tuning of category survival times for our use case (we reference facts from weeks ago regularly).

### Recommendation
**Phase A now, Phase B queue.** Start with manual decay in dream cycle (auto-archive stale proposals, tag memory importance). Evaluate Phase B after OC upgrade is done. Phase C is too risky without more testing.

## Deep Dive 3: Supply Chain Security — Urgent

### What It Is
The npm supply chain threat landscape has escalated dramatically in 2026:
- **Axios compromise** (Mar): 100M+ weekly downloads, North Korean actor, RAT distributed
- **Bitwarden CLI hijack** (Apr): specifically targeted AI coding tool configurations
- **Mini Shai-Hulud** (late Apr): SAP ecosystem packages compromised
- **PyTorch Lightning** (May): credential theft from dev environments
- **Slopsquatting**: AI hallucinated package names being registered as malicious packages

The attack pattern has shifted: attackers now specifically target AI agent configurations (`.claude/settings.json`, `openclaw.json`) for persistence. A compromised dependency could modify our agent behavior without us noticing.

### How It Applies to Us
- We run a Node.js-heavy stack with many npm dependencies
- OpenClaw itself + 50+ skills = large attack surface
- We haven't run `npm audit` in... ever? No record of it.
- Our `.claude/settings.json` and `openclaw.json` contain API keys and model configs
- Claude Code (when used) runs with broad filesystem access
- We store 1Password patterns in TOOLS.md (not the actual secrets, but the retrieval paths)

**Specific risks:**
1. Any compromised OpenClaw dependency could intercept API keys
2. Shai-Hulud-style worm could propagate through our skill installations
3. Slopsquatting could affect skill installs via ClawHub
4. Our OpenClaw version (2026.4.19-beta.2) may have known vulnerabilities patched in newer releases

### Implementation Path
1. **Immediate:** `npm audit` on global OpenClaw install
2. **Immediate:** Review `package-lock.json` for any known compromised packages (Axios, PyTorch Lightning proxies)
3. **Short-term:** Set up `npm audit` as part of daily cron
4. **Short-term:** Verify no unknown MCP servers running: `ps aux | grep mcp`
5. **Medium-term:** Implement dependency pinning for critical packages
6. **Difficulty: trivial (audit) to moderate (ongoing monitoring)**

### Trade-offs
- **Gain**: Peace of mind, defense against escalating threat landscape, responsible practice
- **Cost**: Minimal. `npm audit` is free and fast. Ongoing monitoring adds a cron job.
- **Risk of NOT doing it**: Potentially catastrophic. A compromised dependency could exfiltrate Alex's API keys, 1Password paths, contact information.

### Recommendation
**Do it now.** `npm audit` is a 30-second command. There is no reason not to run it tonight. The supply chain threat is real, escalating, and specifically targeting AI agent setups like ours.
