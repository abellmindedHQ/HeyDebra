## Deep Dive: Anthropic Performance Degradation Post-Mortem

### What It Is
On April 23, Anthropic published a detailed technical post-mortem explaining weeks of user-reported Claude degradation. Three separate product-layer changes (not model weight changes) caused the issues:

1. **Reasoning effort reduction (Mar 4):** Default reasoning effort for Claude Code changed from high→medium to fix UI latency ("frozen" appearance). This caused noticeable intelligence drops on complex tasks.
2. **Caching bug (Mar 26):** A caching optimization was supposed to clear thinking history once after 1 hour of inactivity. Instead, it cleared thinking on EVERY subsequent turn, causing the model to lose short-term memory. This caused repetitive/forgetful behavior.
3. **Verbosity limits (Apr 16):** System prompt instructions limited text between tool calls to 25 words and final responses to 100 words. This caused a 3% coding quality drop in evaluations.

BridgeMind reported Opus 4.6 accuracy dropped from 83.3% to 68.3% during this period. Stella Laurenzo audited 6,852 Claude Code session files showing degraded reasoning depth. Anthropic reverted all three changes and fixed the caching bug in v2.1.116.

### How It Applies to Us
- **We use Opus 4.6 via API, not Claude Code/Cowork.** Anthropic confirmed the API was unaffected. Our quality should have been consistent.
- **Paperclip agents use Claude Code?** Need to verify — if any agents run via Claude Code harness, they may have been affected during Mar 4-Apr 23.
- **The Opus 4.7 vulnerability data (52% from Forbes/Veracode) was collected during the degradation window.** This is CRITICAL — the data that kept us from upgrading may have been measuring the harness bug, not inherent model quality. We should re-evaluate Opus 4.7 with the fixes in place.
- **The caching bug mirrors our own memory issues.** We've had sessions where context seemed lost — this may explain some of those experiences if any path touched Claude Code.

### Implementation Path
1. Verify Paperclip agents' Claude integration path (API vs Code harness) — trivial
2. Re-evaluate Opus 4.7 after fixes settle (~1 week) — moderate
3. If Opus 4.7 evaluations improve post-fix, propose upgrade to Alex — trivial
4. Difficulty: **trivial to moderate**

### Trade-offs
- Upgrading to Opus 4.7 could bring 13% coding improvement and self-verification behavior
- Risk: new model could introduce different behavioral patterns that need learning
- Cost: same pricing as 4.6 (no cost impact)

### Recommendation
**Watch it — re-evaluate in 1 week.** Wait for independent benchmarks post-fix. If the 52% vuln rate was an artifact of the harness bug, Opus 4.7 becomes a clear upgrade.

## Deep Dive: Google ReasoningBank

### What It Is
An open-source memory framework (ICLR 2026) that enables AI agents to learn from both successful and failed experiences during deployment. Key innovation: instead of recording raw action trajectories, it distills higher-level reasoning strategies into structured memory items.

Architecture:
- **Structured memories:** Each item has title, description, and content (distilled reasoning steps/rationales)
- **Closed-loop workflow:** Retrieval → Execution → Self-assessment → Extraction → Consolidation
- **Failure learning:** Actively analyzes failed experiences for counterfactual signals and pitfalls
- **MaTTS (Memory-aware Test-time Scaling):** Allocates extra compute for parallel exploration, using discarded trajectories as learning data

Results: +8.3% success rate on WebArena, +4.6% on SWE-Bench-Verified vs memory-free agents.

### How It Applies to Us
This is essentially what our dream cycle does manually:
- Our `corrections.md` = ReasoningBank's failure distillation
- Our `MEMORY.md` critical lessons = ReasoningBank's structured memories
- Our dream cycle Phase 2 = ReasoningBank's self-assessment + extraction
- Our promotion pipeline (3x corrections → MEMORY.md) = ReasoningBank's consolidation

The difference: ReasoningBank does this automatically and continuously, not nightly. And it uses vector retrieval + structured format for real-time context injection.

**Integration possibility:** Use ReasoningBank as the engine behind our corrections→memory pipeline:
1. Replace manual corrections.md with ReasoningBank memory items
2. Feed session transcripts through ReasoningBank's extraction pipeline
3. Auto-inject relevant memories into session context via retrieval
4. Keep dream cycle for high-level synthesis and human review

### Implementation Path
1. Clone the GitHub repo and evaluate locally — moderate
2. Map our existing corrections.md entries to ReasoningBank format — moderate  
3. Build integration between OpenClaw session context and ReasoningBank retrieval — significant
4. Difficulty: **significant** (but high ROI)

### Trade-offs
- Gain: automated learning from failures, real-time memory injection, reduced dream cycle manual work
- Cost: new dependency, integration complexity, potential over-fitting to past failures
- Maintenance: another service to run and monitor

### Recommendation
**Queue it.** This is the right long-term direction for our memory architecture. Not urgent (our manual system works), but should be a Q2 project. Start with a local evaluation.

## Deep Dive: Neo4j MemMachine

### What It Is
Open-source memory layer for AI agents presented at NODES AI 2026. Uses Neo4j to store and refine agent memories. Builds on Neo4j's Aura Agent platform (GA since Feb 2026).

Key features:
- Stores memories as graph nodes with relationships
- Refines memories over time (consolidation, pruning, linking)
- Supports multi-agent shared memory
- Integrates with Aura Agent for ontology-driven agent construction

### How It Applies to Us
We already have:
- Neo4j running locally (localhost:7474/7687) with SecondBrain data
- Obsidian vault as the human-readable layer
- QMD for hybrid search
- Memory files for agent context

MemMachine could unify these into a single graph-based memory layer where:
- Agent memories (corrections, lessons, patterns) live as Neo4j nodes
- SecondBrain knowledge and agent memory share a single graph
- Multi-agent memory sharing (Debra + Paperclip agents) becomes possible
- Graph relationships surface insights that flat files can't

### Implementation Path
1. Evaluate MemMachine's schema and API — moderate
2. Map existing memory files to graph structure — moderate
3. Build Neo4j-backed memory retrieval for OpenClaw — significant
4. Migrate from file-based to graph-based agent memory — major
5. Difficulty: **significant to major**

### Trade-offs
- Gain: unified memory architecture, graph-based reasoning, multi-agent sharing
- Cost: high complexity, Neo4j dependency (it's been unstable in our setup), migration effort
- Risk: Neo4j has been down since Mar 28 — relying on it for agent memory adds fragility

### Recommendation
**Watch it.** Great concept but premature for us. Neo4j needs to be stable first (it's been down nearly a month). Evaluate after Neo4j is reliably running again.
