## Deep Dive 1: Claude Design — Could It Replace Our Paperclip Design Pipeline?

### What It Is
Claude Design is Anthropic's new design tool (launched ~Apr 20), powered by Opus 4.7's enhanced vision capabilities. It's positioned as a prompts-to-prototypes pipeline: feed it screenshots, prompts, or codebases and it produces interactive prototypes, slide decks, and marketing collateral. Key features:
- Reads existing codebases/mockups to establish brand systems
- Chat + inline comments + direct edits for refinement
- Custom sliders for spacing, color, layout
- Exports to Claude Code (build-ready bundle), Canva, PPTX, PDF, HTML
- Mike Krieger (Anthropic CPO, ex-Instagram) resigned from Figma's board 3 days before launch

### How It Applies to Us
We're running the ABE-32 brand identity through a multi-agent Paperclip pipeline: Kit (mood boards) → Sable (design execution) → Wren (QA) → Maren (thesis alignment). This works but has friction:
- QA issues slip through (typos, eye placement, mixed caps)
- Heartbeats don't always catch new feedback
- Multiple rounds needed for basic corrections
- No direct manipulation — everything is prompt-based

Claude Design could potentially:
1. **Replace or augment the Sable role** — direct manipulation + chat refinement
2. **Improve QA** — Opus 4.7 vision can catch visual errors humans miss
3. **Speed up iteration** — real-time edit sliders vs. prompt-and-wait cycles
4. **Generate export-ready collateral** — business cards, merch mockups directly

### Implementation Path
1. **Trial run (trivial):** Feed current BRAND.md + logo mark into Claude Design, see what it produces
2. **Parallel test (moderate):** Run next mockup round through both Paperclip and Claude Design, compare quality/speed
3. **Integration decision:** If Claude Design is faster and higher quality, restructure Paperclip pipeline to use it for design execution while keeping agents for strategic/QA roles

### Trade-offs
- **Gain:** Faster iteration, direct manipulation, fewer QA failures, export-ready output
- **Lose:** Multi-agent collaborative workflow, Paperclip as a demo/showcase, agent orchestration experience
- **Risk:** Claude Design is brand new — could be unstable, limited, or changed rapidly
- **Cost:** Likely included in Claude subscription or API usage we already pay for

### Recommendation
**Queue it.** Worth evaluating after ABE-32 ships. Don't disrupt the current pipeline mid-project. But for future brand/design work, this could be a major accelerant. Flag to Alex during review.

## Deep Dive 2: MCP STDIO Vulnerability — What This Means for Our Security Posture

### What It Is
OX Security researchers discovered that Anthropic's MCP SDK has a fundamental design flaw in the STDIO transport interface. The STDIO transport is designed to start local servers and pipe I/O, but the command execution is unconstrained — any arbitrary OS command can be executed through the config. This affects:
- All MCP SDK languages (Python, TypeScript, Java, Rust)
- 7,000+ publicly accessible servers
- 150M+ package downloads
- 11 CVEs across major projects (LiteLLM, LangChain, Flowise, Windsurf, Cursor)

Attack vectors:
1. **Command injection via STDIO config** — authenticated or unauthenticated
2. **Zero-click prompt injection** — editing MCP config through injected prompts
3. **MCP marketplace poisoning** — hidden STDIO configs in shared MCP servers

Anthropic's response: "Expected behavior." They've declined to modify the architecture.

### How It Applies to Us
Current exposure assessment:
- **Claude Code:** We run with `--permission-mode bypassPermissions`. While this is the agent permission model (not MCP), it means Claude Code has unrestricted system access. If it connects to any MCP server, that server's config could execute arbitrary commands.
- **OpenClaw MCP:** OpenClaw uses MCP for some integrations. Need to audit which MCP servers are configured and whether any use STDIO transport.
- **Future risk:** Alex has discussed building MCP integrations. Any custom MCP server we build inherits this vulnerability.

### Implementation Path
1. **Audit (trivial):** List all MCP configurations on the Mac mini. Check which use STDIO transport.
2. **Document (trivial):** Add security notes to TOOLS.md about MCP STDIO risks.
3. **Mitigate (moderate):** For any STDIO-based MCP, add input validation. Pin command paths. Don't accept user-controlled command strings.
4. **Monitor (ongoing):** Watch for Anthropic to change their position. Track CVE patches for tools we use.

### Trade-offs
- **Gain:** Understanding of a real attack surface
- **Lose:** Nothing — this is pure security hygiene
- **Risk:** Ignoring it means a compromised MCP server could execute arbitrary commands with our user's permissions
- **Cost:** ~1 hour for audit + documentation

### Recommendation
**Do it now.** The audit is trivial and the risk is real. Especially with the compound finding from last cycle (GitHub agent prompt injection). Our setup runs with high permissions on a machine that has access to Alex's email, contacts, messages, and financial data. Defense in depth.

## Deep Dive 3: GSD Framework for Claude Code — Applicable to Our Agent Workflows

### What It Is
The GSD (Get Stuff Done) Framework structures Claude Code application builds into three distinct phases:
1. **Plan:** Define what to build, architecture decisions, file structure. Output: a plan document.
2. **Execute:** Write code following the plan. Each execution session has a scoped task from the plan.
3. **Verify:** Check that built code matches the plan. Run tests. Flag deviations.

Key insight: Each phase runs in its own context window, preventing "context compounding" where planning exploratory text + code diffs + verification loops fill the window and degrade quality.

### How It Applies to Us
This maps directly to problems we've seen:
- **Paperclip agents losing context:** Sable doesn't always pick up new feedback from heartbeats. The ABE-32 rounds showed context drift (generating "ABOLMINDEA", adding mouths to the eye mark, using placeholder text).
- **Coding agent skill:** Our current coding-agent skill spawns Claude Code/Codex as one long session. For large projects, this leads to the same degradation GSD describes.
- **Dream cycle itself:** The 4-phase structure of dream cycle is essentially a GSD-like pattern (scan → reflect → research → propose), each with defined scope.

### Implementation Path
1. **Coding agent update (moderate):** Modify coding-agent SKILL.md to recommend GSD phases for projects with >5 features or multi-session builds. Add template plan document and verify checklist.
2. **Paperclip agent design (moderate):** Suggest to Alex that Paperclip agents could benefit from explicit plan/execute/verify phases per design round.
3. **Template (trivial):** Create a GSD plan template in workspace for reuse.

### Trade-offs
- **Gain:** Higher quality output from long projects, less context drift, more predictable agent behavior
- **Lose:** More overhead per project (creating plan docs, running verify steps)
- **Risk:** Over-structuring small tasks. Need a "use this when..." threshold.
- **Cost:** 1-2 hours to update coding-agent skill + create templates

### Recommendation
**Queue it.** Good improvement but not urgent. Add to the coding-agent skill during the next skill maintenance pass. The principle is sound and directly addresses observed problems.

## Deep Dive 4: Cloudflare Agent Memory — Lessons for Our Memory Architecture

### What It Is
Cloudflare's Agent Memory is a managed service (private beta Apr 17) that:
- Extracts information from agent conversations automatically
- Stores it in a retrieval-based architecture (not dumped into context)
- Handles temporal logic (facts that change over time)
- Manages supersession (new facts replacing old ones)
- Addresses "context rot" — quality degrades as context windows fill

Key architectural decisions:
- Opinionated API (not raw filesystem access for the model)
- Retrieval over stuffing (surface relevant memories, don't dump everything)
- Managed extraction pipeline (the model doesn't decide what to remember)

### How It Applies to Us
Our memory architecture comparison:
| Feature | Cloudflare Agent Memory | Our Setup |
|---------|------------------------|-----------|
| Extraction | Automatic from conversations | Manual (we write to files) |
| Retrieval | Managed semantic search | memory_search + qmd |
| Temporal logic | Built-in | Manual (we update/overwrite) |
| Supersession | Automatic | Manual (and we often miss it — see stale entries) |
| Context management | Never fills context | We load MEMORY.md + daily notes every session |
| Cost | Managed service pricing | File I/O (free) + Gemini embeddings |

Our biggest gap: **supersession**. When facts change (model version, bug status, project status), we rely on manual updates that frequently go stale. Cloudflare's automatic supersession handling is exactly what we need.

### Implementation Path
1. **Immediate (trivial):** Add supersession awareness to dream cycle Phase 2.5. When memory verification finds stale data, auto-generate the fix (diff preview) so Alex can approve with one click.
2. **Near-term (moderate):** Build a "memory lint" cron that checks MEMORY.md entries against recent daily notes for contradictions. Similar to wiki linting proposal from last cycle.
3. **Long-term (significant):** Evaluate cloud-backed LanceDB memory (mentioned in OC 2026.4.* release notes) as an upgrade path from flat files.

### Trade-offs
- **Gain:** Fresher memory, less stale data, less manual maintenance
- **Lose:** Simplicity of flat files (easy to read, debug, version control)
- **Risk:** Over-engineering memory for a single-user assistant
- **Cost:** Trivial for immediate fix, moderate for memory lint cron

### Recommendation
**Do the trivial fix now, queue the rest.** The supersession gap is real and causes repeated issues. Making the dream cycle auto-generate diffs for stale entries (instead of just flagging them) would cut the fix latency from days to hours.
