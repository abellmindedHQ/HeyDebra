## Deep Dive 1: Anthropic Claude Code Post-Mortem — Implications for Opus 4.7

### What It Is
Anthropic published a detailed engineering post-mortem on Apr 23 explaining that the widely reported Claude Code quality degradation (Mar-Apr 2026) was caused by three product-layer (harness) bugs, not model quality regression:

1. **Reasoning effort downgrade (Mar 4):** Default switched from `high` to `medium` to reduce latency. Users experienced lower intelligence. Reverted Apr 7.
2. **Caching bug (Mar 26):** A session idle optimization was supposed to clear old thinking blocks once after 1 hour of inactivity. Bug caused it to clear thinking on *every turn* for the rest of the session. Claude became "forgetful and repetitive." Also burned through usage limits faster due to constant cache misses. Fixed Apr 10.
3. **System prompt word limit (Apr 16):** Instruction added to limit text between tool calls to 25 words and final responses to 100 words. Killed coding quality. Reverted Apr 20.

Key detail: Opus 4.7 now defaults to `xhigh` reasoning effort. All other models default to `high`. The post-mortem confirms the 52% vulnerability rate reported by Forbes (BridgeMind benchmark dropping 83.3%→68.3%) was almost certainly measured during the degradation period when harness bugs were active.

Anthropic also back-tested their Code Review tool against the offending PRs using Opus 4.7, which caught the bugs — suggesting Opus 4.7's self-verification capabilities are genuine.

### How It Applies to Us
We've been staying on Opus 4.6 since DC#16 flagged the 52% vuln rate. This was the right call at the time — better safe than sorry. But the post-mortem changes the calculus:

- The vulnerability data was collected during the exact window of harness bugs
- Opus 4.7 has `xhigh` default reasoning (better than Opus 4.6's `high`)
- Self-verification improvements directly address our chronic `report-without-verifying` pattern
- Enhanced vision could help with screenshot-based verification workflows
- Claude Code v2.1.116+ has all fixes applied

Our TOOLS.md still references the Claude Code git reset bug from v2.1.87. The post-mortem fixes are in v2.1.116+.

### Implementation Path
1. **Update TOOLS.md** — note v2.1.116+ fixes, update Claude Code version guidance (trivial)
2. **Test Opus 4.7 in sandboxed session** — run a representative workload (coding, messaging, research) and compare quality (moderate)
3. **If tests pass, switch primary model** — update gateway config (trivial, but needs Alex's approval)
4. **Update Paperclip agents** — if Opus 4.7 proves safe, switch all 10 agents (trivial)

### Trade-offs
- **Gain:** Better reasoning (xhigh default), self-verification, enhanced vision, improved coding quality
- **Cost:** Risk of undiscovered issues (Anthropic only tested their own harness bugs, not third-party evaluations). Mythos is gated for a reason — Opus 4.7 has intentionally reduced cybersecurity capabilities vs Mythos
- **Mitigation:** Test in sandbox first. Keep Opus 4.6 as fallback. Don't switch Paperclip until confirmed stable.

### Recommendation
**Queue it.** Test Opus 4.7 next week when Alex is available to approve. Don't rush — we've been fine on 4.6 and the Amsterdam trip is May 4. Stability beats bleeding edge before travel.

## Deep Dive 2: OpenClaw 2026.4.24 — Upgrade Assessment

### What It Is
Major release (published Apr 25) with several categories of changes:

**High-value for us:**
- **Memory-core hybrid search** now exposes raw `vectorScore` and `textScore` alongside combined score. This lets us debug why memory_search returns (or misses) specific results. Currently we can't tell if a miss is vector-side or text-side.
- **Heartbeat timer fix** — clamps oversized scheduler delays that could cause 1ms crash loops. We've seen unexplained heartbeat issues — this might be the cause.
- **Browser automation** — coordinate clicks, 60s default action budget (up from whatever it was), per-profile headless overrides. Helps LinkedIn/social cleanup workflows.
- **MCP session idle eviction** — `mcp.sessionIdleTtlMs` prevents leaked MCP sessions from accumulating. Good hygiene.

**Interesting but not urgent:**
- **Google Meet plugin** — could attend Alex's ORNL meetings, take notes, export artifacts. But requires Chrome + Google auth setup. Medium effort.
- **Real-time voice loops** — Talk/Meet can consult the full agent. Cool for future voice interactions.
- **DeepSeek V4 Flash/Pro** — in catalog as onboarding default. Not relevant unless we want cheaper model options.

**Risk factors:**
- Breaking change: Pi extension factory removed. We don't use Pi, so no impact.
- Users reporting macOS gateway left in mixed-version/unhealthy state after update
- Telegram webhook startup failures on VPS (not applicable to us)

### How It Applies to Us
We're on 2026.4.19-beta.2, now 5 days behind. The hybrid search score transparency is the most immediately valuable feature — it would help us understand and improve memory_search quality. The heartbeat fix could resolve silent issues.

### Implementation Path
1. **Read full changelog** for any additional risks (trivial)
2. **Backup current config** — `cp ~/.openclaw/config/openclaw.json ~/.openclaw/config/openclaw.json.bak` (trivial)
3. **Upgrade:** `npm install -g openclaw@2026.4.24` (moderate risk — macOS mixed-version reports)
4. **Verify services** — gateway restart, confirm BB, browser, crons all work (moderate)
5. **Enable Google Meet plugin** later as a separate project (significant)

### Trade-offs
- **Gain:** Better search debugging, heartbeat stability, browser automation improvements
- **Cost:** Risk of mixed-version state on macOS, ~30 min downtime for upgrade + verification
- **Mitigation:** Backup config, upgrade during off-hours, verify systematically

### Recommendation
**Do it soon.** Upgrade to 2026.4.24 early next week (Mon/Tue). The heartbeat fix and memory search transparency are worth it. Don't do it before Amsterdam trip prep is complete.

## Deep Dive 3: Verification-as-Bottleneck — Applying to Our Workflow

### What It Is
Emerging consensus from HN and dev community that the AI coding maturity ladder is:

1. **Model** — which LLM to use (commoditizing)
2. **Workflow** — how the AI fits into dev process (maturing)
3. **Skills** — reusable project-specific instructions (OpenClaw is ahead here with ClawHub)
4. **Orchestration** — coordinating multiple agents (Paperclip, Frontier)
5. **Verification** — confirming output correctness (THE CURRENT BOTTLENECK)

Key insight: teams using adversarial review (two agents arguing about code quality, red-team skills) catch significantly more bugs than single-model review. This maps directly to our GAN thinking philosophy (SOUL.md).

### How It Applies to Us
Our #1 chronic correction is `report-without-verifying` (6+ occurrences). We also have the broader pattern of Paperclip agents marking tickets "done" without actually completing work (ABE-85 incident).

Currently our verification is manual: "screenshot it before reporting." But this is willpower-based and keeps failing. We need structural verification:

1. **Self-verification step in messaging workflow** — before sending any "it's done" message, auto-run a verification check (load URL, screenshot, compare to expectations)
2. **Adversarial review for Paperclip** — when a ticket is marked "done," have a separate agent (Wren/QA) verify the output before closing
3. **Deploy-and-verify skill** — a reusable skill that deploys, waits, loads the URL, screenshots, and reports pass/fail

### Implementation Path
1. **Add verification checklist to AGENTS.md** — expand the "report-without-verifying" rule with concrete steps (trivial)
2. **Build a deploy-verify skill** — takes a URL, screenshots it, compares to last known good, reports diff (moderate)
3. **Configure Paperclip QA flow** — Wren (CQO) reviews all "done" tickets before they close (moderate, but Paperclip is broken)
4. **Explore adversarial self-review** — before finalizing work, run a "red team" pass asking "what could be wrong?" (moderate)

### Trade-offs
- **Gain:** Drastically reduce report-without-verifying pattern. Catch Paperclip false-dones. Build trust.
- **Cost:** Extra time per task (30s-2min for verification). Extra tokens for adversarial review.
- **Mitigation:** Only verify externally-visible outputs. Skip verification for internal memory/file operations.

### Recommendation
**Do it now (Phase 1).** Add the verification checklist to AGENTS.md this cycle. The deploy-verify skill and Paperclip QA flow can come later.

## Deep Dive 4: Neo4j 2026.04.0 + Aura Agent Platform

### What It Is
Neo4j 2026.04.0 (Apr 23) expands transaction ID length for better concurrency. More importantly, Neo4j Aura Agent platform is now GA — a managed service for building knowledge-graph-powered AI agents with GraphRAG.

GraphRAG = retrieval-augmented generation where the retrieval layer is a knowledge graph, not just vector similarity. This means queries find information based on *relationships* (who knows whom, what's connected to what) rather than just semantic similarity.

### How It Applies to Us
Our SecondBrain architecture already uses Neo4j as the graph layer. But it's been down/underused. Current search is qmd (BM25 + semantic), which misses relationship-based queries like:
- "Who introduced Alex to Brandon Bruce?"
- "What projects connect to Mirror?"
- "Which contacts are in both the ORNL and startup worlds?"

GraphRAG would make these queries natural. The Aura Agent platform could replace our manual Neo4j + qmd setup with a managed solution.

### Implementation Path
1. **Get Neo4j running again** (prerequisite — it's been down since ~Mar 28)
2. **Evaluate Aura Agent vs self-hosted** — pricing, data sovereignty, feature comparison
3. **Build GraphRAG pipeline** — connect SecondBrain vault → Neo4j → agent query layer
4. **Integrate with memory_search** — add graph-based retrieval as a third search mode alongside vector and text

### Trade-offs
- **Gain:** Relationship-aware retrieval, better people/project queries, richer context for conversations
- **Cost:** Neo4j has been unreliable. Aura adds a cloud dependency. GraphRAG pipeline is significant engineering.
- **Mitigation:** Start with self-hosted Neo4j (we have the instance). Evaluate Aura only if self-hosted proves unstable.

### Recommendation
**Watch it.** Get Neo4j running first (that's the real blocker). GraphRAG pipeline is a good project for after Amsterdam trip. Aura Agent evaluation can wait.
