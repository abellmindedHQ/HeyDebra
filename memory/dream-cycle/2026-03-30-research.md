## Deep Dive 1: Long-Running Claude for Scientific Computing

### What It Is
Anthropic's research team published a detailed methodology for multi-day autonomous agent workflows. Siddharth Mishra-Sharma used Claude Opus 4.6 to implement a differentiable cosmological Boltzmann solver (clax) in JAX — a task that typically takes researchers months to years. The key insight: rather than conversational back-and-forth, you specify high-level objectives and let agent teams work autonomously with occasional human oversight.

**Core patterns documented:**
1. **Test Oracle**: A reference implementation that the agent can check its work against automatically. The agent runs its output vs. the oracle continuously, bisecting discrepancies.
2. **Progress File**: A persistent markdown file the agent reads/writes to track what's done, what's next, and what's blocked. Survives across sessions.
3. **Sub-agent Spawning**: Main agent delegates well-scoped subtasks to sub-agents, each with clear success criteria.
4. **SLURM Integration**: Using HPC clusters with job schedulers for compute-heavy tasks.

**Scale:** Claude's C compiler project ran ~2,000 sessions. The Boltzmann solver was deeply coupled (errors cascade), requiring sequential reasoning with selective parallelism.

### How It Applies to Us

We're already doing a lightweight version of this:
- **Progress file** = our `active-context.md` (working memory that persists across sessions)
- **Sub-agent spawning** = OpenClaw's `sessions_spawn` for coding tasks, research, batch processing
- **Test oracle** = we don't have this. GSD reports are generated without validation against ground truth.

**Specific opportunities:**
1. Our GSD agent should have a "test oracle" — cross-reference Things 3 completion status against reported items before generating the report. This would have prevented the Boston hotel false positive.
2. The progress file pattern could improve our coding agent skill. Instead of fire-and-forget sub-agents, have them maintain a progress.md that the parent can monitor.
3. For SecondBrain pipeline: implement oracle-style validation where processed outputs are spot-checked against source data.

### Implementation Path
1. Add oracle validation to GSD agent (check Things 3 + calendar + active-context before reporting) — **moderate effort, high impact**
2. Add progress.md pattern to coding-agent skill SKILL.md — **trivial effort**
3. Implement for voice-notes-processor (compare transcript word count vs. audio duration as sanity check) — **trivial effort**

### Trade-offs
- Oracle pattern requires a reference source. For creative tasks (Batman storybook), there's no oracle. Only useful for data-driven workflows.
- Progress files add token overhead per session (agent reads/writes them each turn).
- Benefit: dramatically reduces cascading errors and stale data problems.

### Recommendation
**Do it now** for GSD agent (addresses known stale data problem). **Queue it** for coding-agent and voice-notes-processor.

## Deep Dive 2: AI Agent Memory Frameworks (Mem0, Zep, Letta, Cognee)

### What It Is
The agent memory landscape has matured significantly in 2026. Four frameworks stand out:

**Mem0**: Dedicated memory layer with multi-level scoping (user/session/agent). Extracts facts from conversations automatically. Hybrid retrieval (vector + metadata filtering). Built-in version control for memories.

**Zep**: Progressive summarization — condenses conversation history while preserving key info. Extracts entities, intents, and facts into structured format. Temporal + semantic search.

**Letta (formerly MemGPT)**: Self-editing memory via virtual context management. The agent manages its own memory (write/delete/update) through tool calls. Hierarchical memory with core memories (always loaded) vs. archival (searched on demand).

**Cognee**: Builds knowledge graphs from conversations. Structures unstructured data into graph relationships. Cross-references across sessions and users.

### How It Applies to Us

Our current memory architecture:
| Our Layer | Framework Equivalent | Gap |
|-----------|---------------------|-----|
| active-context.md | Letta core memory | We do this manually; Letta does it via tool calls |
| memory/YYYY-MM-DD.md | Zep progressive summaries | We write raw logs; Zep auto-summarizes |
| MEMORY.md | Mem0 long-term facts | We curate manually; Mem0 auto-extracts |
| memory_search | Vector retrieval | Works but basic — no temporal weighting |
| SecondBrain/Neo4j | Cognee knowledge graph | Similar concept, but disconnected from agent memory |

**Key gaps in our system:**
1. **No automatic fact extraction.** When Alex says "Hannah is pregnant" in conversation, we manually write it to MEMORY.md. Mem0 would extract and store it automatically.
2. **No temporal weighting.** memory_search returns results by semantic similarity, not recency. A fact from Day 1 ranks equally with yesterday's update.
3. **No progressive summarization.** Daily memory files grow indefinitely. We never condense March 28's 300-line file into a 20-line summary.
4. **Memory is disconnected from Neo4j.** Our people graph (SecondBrain) doesn't feed into agent memory search.

### Implementation Path
1. **Add temporal decay to memory_search** — weight recent memories higher. Would require OpenClaw core changes or a wrapper. **Significant effort, medium impact.**
2. **Build auto-summarization into heartbeat** — periodically condense daily files older than 3 days into summaries. **Moderate effort, high impact.** Could do this with a simple cron.
3. **Bridge Neo4j → memory** — when memory_search misses, fall back to Neo4j entity lookup. **Significant effort, high impact.** Requires Neo4j to be running (it's down).
4. **Adopt Mem0 for auto-extraction** — would replace our manual MEMORY.md updates. **Major effort, uncertain benefit.** Our manual approach gives us control over what's remembered.

### Trade-offs
- Auto-extraction risks storing noise (every casual mention becomes a "fact")
- Progressive summarization risks losing nuance (the detail about Brandon Brown's wife's name might get compressed away)
- Neo4j bridge adds a dependency (it's already been down for 3 days)
- Our current manual approach is labor-intensive but high-quality

### Recommendation
**Do it now:** Auto-summarization of old daily files (low risk, high value). **Queue it:** Neo4j bridge (needs Neo4j stable first). **Watch it:** Mem0/Letta adoption (our manual approach works well enough for now). **Skip:** Temporal decay in memory_search (requires OC core changes, diminishing returns).

## Deep Dive 3: Claude Opus 4.6 Capabilities We're Not Using

### What It Is
Claude Opus 4.6 brought several capabilities we're paying for but not fully leveraging:

1. **1M token context window**: We can process massive documents in a single session. Our sub-agents typically work with much smaller contexts.
2. **Computer use improvements (72.5% OSWorld)**: Dramatically better at interacting with desktop environments — clicking, typing, navigating screens.
3. **Claude Code hooks**: Custom pre/post hooks on tool calls. Can enforce patterns like "always verify before send."
4. **Remote continuity**: Claude Code sessions can survive interruptions and resume.

### How It Applies to Us

**1M context window — immediate opportunities:**
- Process Alex's entire Apple Notes export in a single pass (200+ notes) instead of batching
- Feed the entire SecondBrain vault index to the LLM export processor for richer cross-referencing
- Load all 15 pending dream cycle proposals + their context in one shot for Alex's review session

**Computer use — current usage:**
- LinkedIn inbox cleanup (browser automation) — already using this
- Could expand to: YNAB budget review, Monarch transaction monitoring, Things 3 web UI for tasks that CLI can't handle

**Claude Code hooks — not using at all:**
- Could implement a pre-send hook that validates recipient before any message tool call
- Could implement a post-write hook that runs linting on generated code
- This directly addresses our "wrong recipient" pattern

**Remote continuity — not using:**
- Long-running coding tasks currently fire-and-forget via sub-agents
- Remote continuity would let us resume a coding session after a gateway restart

### Implementation Path
1. **Add pre-send validation hook** to message tool calls — verify recipient matches intent. **Moderate effort, HIGH impact** (addresses top failure pattern).
2. **Use 1M context for proposal review** — bundle all 15 proposals with full context for Alex's review. **Trivial effort.**
3. **Expand browser automation** to YNAB/Monarch for financial monitoring. **Moderate effort, medium impact.**
4. **Explore Claude Code hooks** for coding-agent quality gates. **Moderate effort, medium impact.**

### Trade-offs
- Pre-send hooks add latency to every message send (small but real)
- 1M context costs more tokens per request
- Browser automation for financial apps requires login session management
- Hooks require understanding OpenClaw's hook system (if it exists at the gateway level)

### Recommendation
**Do it now:** Pre-send recipient validation (even without hooks — can be a self-imposed checklist in AGENTS.md). **Do it now:** 1M context for proposal review. **Queue:** Financial app browser automation. **Watch:** Claude Code hooks for coding quality.
