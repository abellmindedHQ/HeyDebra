## Deep Dive: NousResearch Hermes Agent

### What It Is
Hermes is a self-improving AI agent with a built-in learning loop. Key differentiator from OpenClaw: **skills self-improve during use** and the agent **autonomously creates new skills after complex tasks** — no nightly dream cycle required. It runs on Telegram, Discord, WhatsApp, Signal, and CLI from a single gateway. Supports any model via OpenRouter (200+ models). Multi-channel conversation continuity, FTS5 session search with LLM summarization for cross-session recall, and Honcho dialectic user modeling (builds a deepening model of who you are). Compatible with the agentskills.io open standard. Runs on $5 VPS to GPU cluster, with serverless persistence (Daytona/Modal).

### How It Applies to Us
Hermes is doing what OpenClaw does but with a tighter learning loop. Our comparison:

| Feature | Debra (OpenClaw) | Hermes |
|---|---|---|
| Multi-channel | ✅ BB/WhatsApp/Discord | ✅ TG/Discord/WA/Signal/CLI |
| Skills | ✅ SKILL.md system | ✅ agentskills.io compatible |
| Memory | ✅ MEMORY.md + QMD | ✅ FTS5 + LLM summaries |
| Self-improvement | 🟡 Nightly cron (manual) | ✅ Autonomous after each task |
| User modeling | 🟡 USER.md (manual) | ✅ Honcho dialectic (automated) |
| Multi-model | 🟡 Manual override | ✅ Native (OpenRouter) |
| Voice | ✅ ElevenLabs TTS | ✅ Voice memo transcription |
| Subagents | ✅ sessions_spawn | ✅ Parallel subagents |
| Cron | ✅ 18 crons | ✅ Built-in |

The gap is in **automated self-improvement**. Hermes closes its learning loop within a session. Our loop runs once per night via dream-cycle. For one-off knowledge/skill captures, we're slower.

**Honcho user modeling** is particularly interesting. Rather than manually updating USER.md and MEMORY.md, Hermes builds a probabilistic model of the user through dialectic exchange — asking clarifying questions and updating beliefs. This is architecturally more scalable than our manual capture approach.

### Implementation Path
**Short-term (no infrastructure changes):**
- Adopt Hermes's approach of creating a new skill file after any complex multi-step task (not just nightly). During Phase 4 coding sessions, if I develop a novel workflow, immediately write a skill.
- After complex tasks: write `memory/YYYY-MM-DD-<task>-learned.md` capturing what worked, what failed, and the pattern.

**Medium-term (new skill):**
- Build an "after-action" skill that fires after complex task completion, captures workflow + lessons, and proposes skill file creation.
- Evaluate agentskills.io compatibility — does OpenClaw support the standard? If so, Hermes skills could be imported.

**Long-term:**
- Evaluate Hermes as a complementary system for Alex's HeyDebra / HeyAvery product vision — Hermes's multi-model / serverless approach could be the backend, OpenClaw the orchestration layer.

### Trade-offs
- **Gain:** Faster skill capture, potentially more accurate user modeling, multi-model flexibility
- **Cost:** Would require building new automation around after-action capture; Honcho integration is non-trivial; Hermes is a competing product (using it means splitting attention/investment)
- **Risk:** Automated skill improvement can degrade skill quality if not carefully constrained — dream cycle's human-review gate exists for good reason

### Recommendation
**Watch it closely.** Steal the after-action capture concept immediately (trivial). The Honcho user modeling and agentskills.io standard are worth deeper evaluation in 30 days.

## Deep Dive: Claudian — Claude Code Inside Obsidian

### What It Is
Claudian is an Obsidian plugin that embeds Claude Code (and Codex) directly in the vault. Your vault becomes the agent's working directory — it reads, writes, searches, and edits notes directly. Features: inline edit with word-level diff, slash commands + Skills, @mention for files/subagents/MCP servers, Plan Mode (agent plans before implementing), MCP server connections, multi-tab conversations. Install via BRAT (auto-updates from GitHub) or manual plugin install.

### How It Applies to Us
Alex's SecondBrain (~/SecondBrain/) is his Obsidian vault. Currently our interaction model is:
- Debra → `qmd` CLI → keyword/semantic search → results returned as text
- Debra → Neo4j → graph queries → relationship data
- Alex → Obsidian → reads/edits manually

Claudian would add: **Alex → chat in Obsidian → Claude Code executes → vault edits**

Use cases that would immediately benefit:
1. **Meeting notes processing:** Alex drops a meeting note, asks "extract action items and update the project page" — happens in-vault without switching to terminal
2. **Knowledge graph building:** Ask Claudian to create wikilinks between related notes
3. **Reflection writing:** Alex journals, asks "what patterns do you see from this week's reflections?" — Claudian reads the last 7 reflection files and synthesizes
4. **Mirror feature:** This is literally the "know yourself" product — a conversational interface over Alex's life data that lives inside the data

The @mention for MCP servers is key — Claudian could call external MCPs for calendar/contacts data while operating inside the vault.

### Implementation Path
**Difficulty: trivial** (just a plugin install)

1. Install Claude Code CLI (already installed per TOOLS.md)
2. In Obsidian: Settings → Community Plugins → Browse → search "Claudian" OR use BRAT
   - BRAT install: Add beta plugin from `https://github.com/YishenTu/claudian`
3. Enable plugin, configure Claude Code path
4. Test: open chat sidebar, ask it to read a People note and summarize

**Caution:** Claudian gives Claude Code direct write access to the SecondBrain vault. Need to make sure:
- Plan Mode is ON for any writes (review before execute)
- Alex is aware that Claude Code (not just Debra) will be making edits
- Backup / git sync for SecondBrain should be active before enabling

### Trade-offs
- **Gain:** Conversational SecondBrain interface Alex has been building toward; no CLI switching; direct in-vault editing with diff preview
- **Cost:** Claude Code subscription charges (or API key needed); adds another AI layer inside Obsidian
- **Risk:** Claude Code has write access to the vault — one bad prompt could do broad damage. Plan Mode mitigates this but doesn't eliminate it.

### Recommendation
**Do it now.** This is the clearest win in tonight's scan. Low effort, directly serves Alex's life's work vision. Propose explicitly in Phase 4.

## Deep Dive: Karpathy Coding Principles (via andrej-karpathy-skills)

### What It Is
A distilled CLAUDE.md (or Claude Code plugin) based on Andrej Karpathy's observations about LLM coding failures. Four principles:
1. **Think Before Coding:** State assumptions, surface ambiguity, push back when warranted, stop when confused
2. **Simplicity First:** Minimum code, no speculative features, no premature abstraction
3. **Surgical Changes:** Touch only what you must, don't "improve" adjacent code, clean up only your own mess
4. **Goal-Driven Execution:** Transform tasks into verifiable goals with success criteria → LLM loops until verified

### How It Applies to Us
Karpathy's diagnosis is accurate for how I operate too — not just Claude Code. Specifically:
- **Wrong assumptions without checking:** I've done this (asking Alex about Boston trip dates that were in memory; assuming cron reminder = authorization)
- **Overcomplicated solutions:** Dream cycle proposals sometimes propose complex infrastructure changes when a simple script would do
- **Orthogonal edits:** When editing skill files, I've changed adjacent sections that weren't part of the request
- **Weak success criteria:** When Alex says "fix the BB attachment thing," the success criteria aren't defined — this leads to underdefined loops

The four principles apply to **how Debra operates**, not just how she codes. They could be adapted into a "before-I-act" checklist.

### Implementation Path
**For Claude Code (immediate):**
```bash
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills
```
This adds the principles to Claude Code globally.

**For Debra (AGENTS.md update):**
Adapt the four principles into a pre-action checklist in AGENTS.md:
- Before any complex task: state assumptions explicitly
- Before any write: verify I understand the scope (surgical)
- Before any proposal: define success criteria, not just "do X"
- Simplicity: if I'm writing >200 lines of anything, ask if there's a shorter path

### Trade-offs
- **Gain:** Higher quality outputs, fewer orthogonal errors, better failure-mode handling
- **Cost:** Slightly slower (more upfront planning), requires discipline to not skip the checklist
- **Risk:** Can't apply to all situations; some tasks are genuinely exploratory

### Recommendation
**Queue for AGENTS.md update.** The four principles are worth adding as a "before you act" section. Low effort, high leverage.

## Deep Dive: Project Glasswing — Security Implications

### What It Is
Anthropic's Claude Mythos Preview (unreleased frontier model) can find thousands of high-severity vulnerabilities in any major OS or browser. Project Glasswing is a defensive coalition ($100M credits, 40+ orgs, 12 founding partners) using Mythos for proactive security scanning. The initiative exists because Anthropic believes offensive AI capability at this level will proliferate — they're trying to front-run it for defense.

### How It Applies to Us
**Short-term (our setup):**
- Our Mac mini + OpenClaw setup has never been security audited. If Mythos-class models can find vulns in major OS components, our custom automation scripts (LaunchAgents, bash scripts with API keys, SSH configs) are not immune.
- The biggest risks in our setup: plaintext API keys in config files, LaunchAgent plist files run as root, the BB transformer proxy running on localhost without auth, the Neo4j instance (password-protected but local network).
- None of these require Mythos to audit — Claude Sonnet could probably surface most of these. We should do a self-audit.

**Medium-term (Alex's product vision):**
- If Anthropic releases Mythos or a Mythos-lite for general use, it could be a game-changer for the security layer of the HeyDebra / HeyX platform.
- Privacy and security are the #1 concern for a personal AI companion product. Being able to say "our infrastructure is Glasswing-audited" could be a differentiator.

**Long-term:**
- Glasswing is Anthropic signaling that the frontier is moving fast and security implications are escalating. The pace of AI capability development is compressing. For an AI-heavy setup like ours, this means: keep infrastructure minimal and auditable, don't accumulate technical debt in security-sensitive scripts.

### Implementation Path
**Self-audit (immediate, trivial):**
1. Run `healthcheck` skill to review current security posture
2. Grep all workspace scripts for hardcoded credentials
3. Review LaunchAgent plist files — what's running as what user?
4. Confirm Neo4j has strong password, localhost-only binding

### Trade-offs
- **Gain:** Reduced risk from our automation surface area
- **Cost:** Time for audit (~1hr)

### Recommendation
**Queue it.** Schedule a healthcheck run as part of next GSD cycle. Not urgent enough to block tonight's proposals, but it's been too long since we audited our automation surface.
