## Deep Dive 1: Claude Code v2.1.94 + Issue #42796 (Usability Regression)

### What It Is
Claude Code released v2.1.94 on April 7 with major bug fixes, especially around rate-limiting (429 errors now surface immediately instead of silent wait) and plugin skill naming. However, a parallel issue (#42796) with 1,294 upvotes indicates a broader usability regression: users report Claude Code has become "unusable for complex engineering tasks" since February 2026 updates. Issues include: ignoring instructions, making "simplest fixes" that are wrong, doing the opposite of requested activities, and claiming completion when not done.

### How It Applies to Us
We delegate coding work to Claude Code via the `coding-agent` skill (specified in OpenClaw SKILL.md). If Claude Code is regressing on complex tasks, our delegated work quality could degrade. Current status: we use Claude Code for (1) building new features, (2) PR reviews, (3) refactoring large codebases, (4) iterative coding. The regression appears real and reproducible per issue #42796. The April 7 v2.1.94 release fixes rate-limiting but does NOT address the instruction-following regression.

### Implementation Path
1. **Test current workflow**: Run coding-agent skill on a known-good task (medium complexity, 10-20 min work). Capture outputs, review for instruction adherence.
2. **Monitor issue #42796**: Watch for Anthropic response or regression fix. If v2.1.95+ ships with instruction fixes, test again.
3. **Fallback strategy**: If Claude Code regresses confirmed, shift complex delegation to Opus 4.6 direct API calls (lower automation but more reliable on instruction-following).
4. **Document**: Add note to TOOLS.md about Claude Code version requirements + fallback procedure.

**Difficulty:** Moderate (requires testing + decision tree). **Time estimate:** 1-2 hours investigation + 30 min decision.

### Trade-offs
**Gain:** Accurate delegation, trust in output quality.
**Cost:** May lose Claude Code's convenience (auto-apply edits, streaming feedback). Direct API approach is more manual but more predictable.

### Recommendation
**Do it within a week.** This is not an immediate blocker (we have workarounds) but is affecting quality of delegated work. Monitor issue #42796 for Anthropic's response. If regression persists past v2.1.95, shift to fallback strategy.

## Deep Dive 2: Project Glasswing + Claude Mythos

### What It Is
Anthropic announced Project Glasswing, a collaborative initiative with 10 major tech companies + Linux Foundation, to secure critical software using AI. The linchpin is Claude Mythos Preview, an unreleased frontier model that can find vulnerabilities "surpassing all but the most skilled humans." Mythos has already discovered thousands of 0-days in every major OS (Windows, macOS, Linux) and web browser (Chrome, Firefox, Safari). The initiative pivots this capability toward defensive use: hardening critical infrastructure rather than exploiting it.

### How It Applies to Us
Claude Mythos is a strategic signal about future Claude capabilities. Today, we use Sonnet 4.6 (general purpose) and Opus 4.6 (reasoning) for most work. When Mythos is released (timeline unknown, possibly 6-18 months), it will be significantly more capable at code analysis, vulnerability detection, and security audits. Our code review skill + codebase refactoring workflows could see major quality improvements. Additionally, Glasswing's defensive security focus aligns with OpenClaw's privacy/security goals (no exfiltration, safe tooling, audit-friendly).

### Implementation Path
1. **Watch Anthropic announcements** for Mythos release date + pricing.
2. **Prepare integration**: When available, create a dedicated `mythos-code-review` skill for security-focused code analysis. Use Mythos specifically for vulnerability scanning, not general tasks (to optimize for cost + accuracy).
3. **Audit our codebase now**: Run current code review on ORNL work, voice notes pipeline, OpenClaw skills. Establish a baseline of issues before Mythos arrives.
4. **Plan for cost**: Mythos will likely be more expensive than Opus. Budget accordingly for security audit runs (possibly quarterly or per-release).

**Difficulty:** Low (watch + wait). **Time estimate:** 20 min for monitoring setup, 4-6 hours for integration when Mythos ships.

### Trade-offs
**Gain:** Future access to frontier-model security scanning. Better vulnerability detection. Alignment with Anthropic's safety mission.
**Cost:** Potential API cost increase. Requires integration work. May take 6+ months for release.

### Recommendation
**Watch it.** This is a strategic bet on Anthropic's roadmap. No action needed now, but add "Glasswing release watch" to cron jobs or monthly review checklist. When Mythos lands, prioritize integration for code security audits.

## Deep Dive 3: Neo4j 5.x Graph Performance Optimization

### What It Is
Neo4j released performance improvements for large-scale graphs (100K+ nodes), including better Cypher query planning, index optimization strategies, and memory tuning. Relevant for knowledge graphs at scale (our use case).

### How It Applies to Us
Our Neo4j instance started at ~2K nodes (SecondBrain people/projects/concepts graph + voice notes sidecars). As voice notes import continues and social data processor adds more relationships, we'll likely hit 10-20K nodes within 6 months. Neo4j 5.x optimizations (specifically Cypher planner improvements + composite indexing) will help us stay performant as scale increases.

### Implementation Path
1. **Current inventory**: Measure current Neo4j instance size (nodes, relationships, query time for slow queries).
2. **Profile slow queries**: If any graph queries take >500ms, profile them with EXPLAIN plan.
3. **Apply v5.x guidance**: Create composite indexes on frequently queried paths (e.g., People → Projects → Meetings). Update Cypher queries to use the newer planner hints.
4. **Upgrade path**: Test on a staging clone of the graph. Measure improvement. If good, upgrade production.

**Difficulty:** Moderate (requires profiling + optimization knowledge). **Time estimate:** 4-6 hours.

### Trade-offs
**Gain:** Better query performance. Supports 10x graph scaling without latency. Future-proofs infrastructure.
**Cost:** Testing overhead. Risk of breaking queries during upgrade (mitigated by staging). Time investment now.

### Recommendation
**Queue for next sprint.** Not urgent (current graph is small enough), but worth doing before voice notes import adds significant scale. Would pair well with the night-swimming-contacts skill (which should use Neo4j for contact enrichment queries).
