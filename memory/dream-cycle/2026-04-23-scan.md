# Dream Cycle #17 — Research Scan (2026-04-23)

### 1. [Claude Mythos Unauthorized Access — Third-Party Breach](https://www.forbes.com/sites/timkeary/2026/04/23/alleged-claude-mythos-breach-raises-questions-about-ai-security/)
**Source:** Forbes/Inc/Gizmodo | **Relevance:** high
**Summary:** A Discord group gained unauthorized access to Anthropic's restricted Claude Mythos Preview model (announced Apr 7 under Project Glasswing). Access came through a third-party contractor who combined privileged access with leaked Mercor data. Anthropic says no core systems were breached.
**Why it matters for us:** We run Anthropic models heavily. This isn't a direct threat to us but it's a signal about supply-chain security in the AI ecosystem. Also, Mythos has exceptional vuln-finding capabilities. If it leaks wider, the threat landscape changes.

### 2. [Opus 4.7 Generates Vulnerable Code — 52% Vulnerability Rate (Forbes/Veracode)](https://www.forbes.com/sites/the-wiretap/2026/04/22/anthropics-claude-is-pumping-out-vulnerable-code-cyber-experts-warn/)
**Source:** Forbes (Apr 22) | **Relevance:** high
**Summary:** Veracode tested Opus 4.7 on 80 coding tasks: 52% of generated code included vulnerabilities (up from 51% on 4.1, 50% on Sonnet 4.5). TrustedSec CEO confirms quality dropped sharply after Opus 4.6 release. OpenAI models scored ~30% vulnerable.
**Why it matters for us:** Validates our decision to stay on Opus 4.6. Do NOT upgrade. This is the second cycle confirming this. The Paperclip team uses Claude for code generation too — their output may need security auditing.

### 3. [OpenClaw 2026.4.22 — Major Platform Update](https://github.com/openclaw/openclaw/releases)
**Source:** GitHub Releases | **Relevance:** high
**Summary:** Big release: xAI image/TTS/STT support, local embedded TUI mode, OpenAI native web_search, `/models add` from chat, WhatsApp per-group systemPrompt config, Active Memory plugin improvements, diagnostics export, Tencent Cloud provider, GPT-5 prompt overlay.
**Why it matters for us:** We're on 2026.4.19-beta.2. The WhatsApp per-group systemPrompt is relevant for our group chat behavior config. Active Memory plugin could replace our manual memory_search pattern. Diagnostics export useful for debugging. Worth upgrading this weekend.

### 4. [Neo4j Agent Memory Library — Three-Layer Graph Memory](https://neo4j.com/labs/agent-memory/)
**Source:** Neo4j Labs | **Relevance:** high
**Summary:** Open-source Python library providing short-term (conversations), long-term (entities/preferences), and reasoning (tool traces) memory layers, all in Neo4j. Integrates with LangChain, PydanticAI, LlamaIndex. Multi-stage entity extraction with spaCy + GLiNER + LLM. Wikipedia enrichment, geocoding, geospatial queries.
**Why it matters for us:** We already have Neo4j running (localhost:7474). This library maps almost perfectly to our SecondBrain architecture. The reasoning memory layer (traces of how problems were solved) could help with our corrections pattern. Worth a deep dive.

### 5. [MCP STDIO Vulnerability — Now 7K+ Servers, Multiple CVEs](https://thehackernews.com/2026/04/anthropic-mcp-design-vulnerability.html)
**Source:** The Hacker News / OX Security | **Relevance:** high
**Summary:** The MCP STDIO architectural flaw now has 11+ CVEs across major projects (LiteLLM, Flowise, Windsurf, etc.). Anthropic calls it "expected behavior" and won't fix. 150M+ downloads affected. New CVEs: Go MCP SDK DNS rebinding (CVE-2026-34742, patched in 1.4.0), Mobile MCP RCE (CVE-2026-35394).
**Why it matters for us:** We use MCP (Kapture MCP on port 61822). We need to verify our MCP setup isn't exposed beyond localhost. The "won't fix" from Anthropic means this is permanent — developers must sanitize.

### 6. [HN Discussion: "AI Agents Don't Understand What They're Building"](https://news.ycombinator.com/item?id=47871703)
**Source:** Hacker News (Apr 23) | **Relevance:** medium
**Summary:** Popular HN thread about the limits of AI coding agents. Codex boosts productivity for simple tasks but makes "mangled, halfway-hallucinated" progress on complex problems. Key insight: agents lack the heuristics to identify their own confusion. Human judgment needed for what agents should remember.
**Why it matters for us:** Validates our Paperclip experience — agents don't self-wake, don't understand brand vision, need constant supervision. Also relevant to memory architecture: "human judgment in deciding what an agent should remember" is exactly our active-context.md pattern.

### 7. [HN: "AI Agent Has Amnesia" — Memory vs Context Pressure](https://news.ycombinator.com/item?id=47866304)
**Source:** Hacker News (Apr 22) | **Relevance:** medium
**Summary:** Discussion about the tension between memory systems and context window pressure. Tools like Codex/Claude perform worse as context fills with skills, memory, plugins, and codebases. Need for "adhoc memory" — dynamic retrieval like human cognition rather than pre-loading everything.
**Why it matters for us:** Directly relevant to our memory architecture. We pre-load MEMORY.md + AGENTS.md + SOUL.md + USER.md + TOOLS.md every session. That's a LOT of context. The OpenClaw Active Memory plugin (message/recent/full modes) could help us be more selective.

### 8. [Microsoft Agent Framework 1.0 — Unifies AutoGen + Semantic Kernel](https://devblogs.microsoft.com/agent-framework/microsoft-agent-framework-version-1-0/)
**Source:** Microsoft DevBlog (Apr 3) | **Relevance:** medium
**Summary:** Microsoft shipped Agent Framework 1.0, merging AutoGen and Semantic Kernel into one SDK. Stable APIs, .NET + Python, long-term support. AutoGen/SK get maintenance only. Targets enterprise multi-agent workflows.
**Why it matters for us:** Interesting as a comparison point for Paperclip's multi-agent approach. Microsoft is betting on a single unified framework vs the fragmented ecosystem. Not immediately actionable but worth monitoring if we ever outgrow Paperclip.

### 9. [OpenClaw Active Memory Plugin — Automated Context Retrieval](https://docs.openclaw.ai/concepts/active-memory)
**Source:** OpenClaw docs / Reddit | **Relevance:** high
**Summary:** Optional plugin (since 2026.4.10) that adds a memory sub-agent before each response. Three modes: message (current only), recent (recent history + memory), full (everything). Requires explicit opt-in in openclaw.json. Can debug with /verbose.
**Why it matters for us:** Could dramatically improve our memory consistency. We keep hitting "search memory before asking Alex" corrections. This plugin automates that. The "recent" mode seems ideal — balances token cost with context freshness. Should evaluate token impact before enabling.

### 10. [GraphRAG + Knowledge Graphs as "Navigation System" for AI Agents](https://beam.ai/agentic-insights/5-ways-knowledge-graphs-are-quietly-reshaping-ai-workflows-in-2026)
**Source:** Multiple (Beam AI, Neo4j blog, Medium) | **Relevance:** medium
**Summary:** GraphRAG is maturing — combining vector search with graph traversal for grounded LLM responses. "Agentic Knowledge Graphs" are emerging where agents dynamically build graph structure as they reason. Multi-agent collaboration through shared graph memory gaining traction.
**Why it matters for us:** Our Neo4j + QMD + Obsidian stack is already a proto-GraphRAG system. The "reasoning memory" concept (recording decision traces) could help with our corrections pattern — instead of just logging what went wrong, log the full reasoning chain that led to the mistake.
