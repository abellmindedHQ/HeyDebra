## Deep Dive 2: Mini Shai-Hulud Supply Chain Campaign

### What It Is
Third wave of the Shai-Hulud npm supply chain campaign by TeamPCP (likely Russia/CIS-based, payload exits on ru locale). This wave specifically compromised SAP packages but the attack vector is universal:

**Attack chain:**
1. Compromise a maintainer's npm token or GitHub account
2. Inject malicious `preinstall` script into legitimate package
3. Script downloads Bun runtime, executes 11.6MB credential stealer
4. Stealer harvests: npm tokens, GitHub PATs, GitHub Actions secrets, AWS/Azure/GCP/K8s secrets
5. Stolen creds encrypted with RSA-4096 + AES-256-GCM, exfiltrated to repos created on victim's own GitHub ("A Mini Shai-Hulud has Appeared")
6. **WORMABLE:** Uses stolen npm tokens to auto-publish itself to every package the token can reach

**New in this wave:**
- **Targets AI coding agents:** Commits `.claude/settings.json` with SessionStart hook and `.vscode/tasks.json` with `runOn: folderOpen` to every accessible repo. Opening repo in VSCode or Claude Code silently re-executes malware. First supply chain attack targeting AI agent configs as persistence vector.
- **Encrypted exfil:** Defenders can't read stolen data even if they find the dead-drop repos.

### How It Applies to Us
- Mac mini has: npm (OpenClaw), 1Password CLI with auto-unlock, SSH keys, GitHub credentials, API keys
- We use Claude Code for coding tasks (SKILL.md coding-agent)
- `.claude/settings.json` persistence vector is DIRECTLY relevant
- We haven't run `npm audit` in... ever?
- 1,800+ repos created with stolen creds = this is actively happening at scale

### Implementation Path
1. **Immediate:** Run `npm audit --global` to check for known vulnerabilities
2. **Immediate:** Check for suspicious `.claude/settings.json` or `.vscode/tasks.json` in any repos
3. **Short-term:** Set up weekly `npm audit` cron
4. **Short-term:** Review npm token scoping (limit to read-only where possible)
5. **Medium-term:** Consider `npm install --ignore-scripts` for untrusted packages

**Difficulty:** trivial (audit) to moderate (ongoing monitoring)

### Trade-offs
- **Gain:** Catch compromised packages before they steal credentials
- **Cost:** Minimal. `npm audit` takes seconds.
- **Risk of inaction:** HIGH. A compromised package could exfiltrate 1Password tokens, GitHub PATs, and all API keys.

### Recommendation
**Do it now.** Run the audit tonight. Set up the cron this week.

## Deep Dive 4: GraphRAG Convergence with SecondBrain

### What It Is
GraphRAG (Graph-based Retrieval-Augmented Generation) is becoming the industry standard for grounded AI. The pattern: combine knowledge graph structure (entities, relationships) with vector embeddings and LLM generation for more accurate, attributable responses.

Key developments:
- **Mem0g:** Graph memory extension of Mem0. Claims improved accuracy on multi-hop questions by storing facts as connected graph nodes rather than flat embeddings.
- **AWS Quick:** Desktop agent that builds a persistent personal knowledge graph and uses it to proactively trigger actions. This is essentially what our dream-cycle + GSD + SecondBrain pipeline does manually.
- **Hybrid architectures:** Vector + graph + file memory becoming standard. This IS our stack (qmd embeddings + Neo4j graph + Obsidian files).

### How It Applies to Us
Our SecondBrain architecture (Neo4j + Obsidian + qmd) is already a hybrid knowledge system. But:
- **Neo4j has been down for 34 days.** The graph layer is completely offline.
- **qmd does vector+BM25 search** but doesn't leverage graph relationships for multi-hop reasoning.
- We're doing manually what AWS Quick automates (proactive triggers from knowledge graph).

The industry is validating our architecture. We just need to actually run it.

### Implementation Path
1. **Immediate:** Restart Neo4j (it's just down, not broken)
2. **Short-term:** Integrate qmd with Neo4j for hybrid retrieval (vector search + graph traversal)
3. **Medium-term:** Evaluate Mem0g as potential upgrade to our memory_search pipeline
4. **Long-term:** Build proactive trigger system (like AWS Quick) that surfaces relevant knowledge before Alex asks

**Difficulty:** trivial (restart Neo4j) to significant (full GraphRAG pipeline)

### Trade-offs
- **Gain:** Multi-hop reasoning over SecondBrain, proactive knowledge surfacing
- **Cost:** Neo4j restart is free. GraphRAG pipeline is a project.
- **Risk:** Low for restart. Medium for architectural changes.

### Recommendation
**Restart Neo4j now. Queue GraphRAG integration for later.** The graph being down for 34 days is embarrassing given that the industry is converging on exactly this architecture.
