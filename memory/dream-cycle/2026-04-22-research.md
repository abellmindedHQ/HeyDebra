1. **Opus 4.6 degradation risk**: If Anthropic silently reduces thinking effort on 4.6 (as they did with 4.7's predecessor), our code quality could degrade without us noticing.

4. **Competition signal**: OpenAI at 30% vs Claude at 52% is a stark difference. If Alex is evaluating models for HeyDebra/Abellminded platform, this is relevant data.

### Implementation Path
1. **Do NOT upgrade to Opus 4.7 yet.** Stay on 4.6.
2. Run a code security scan on Paperclip-generated code (abellminded repos). Difficulty: moderate.
3. Set up a simple benchmark: 10 coding tasks, compare 4.6 vs 4.7 output quality. Difficulty: moderate.
4. Monitor Anthropic's response to the Forbes article for patches/fixes.

### Trade-offs
- Staying on 4.6 means missing the genuine improvements in vision and instruction-following that 4.7 offers
- The vulnerability rates may be inflated by testing methodology (Veracode's 80-task benchmark may not represent our use cases)
- We're API users, not Pro subscribers, so the thinking effort reduction may not apply to us

### Recommendation
**Watch it.** Do NOT upgrade. Monitor for Anthropic's response. Run a security scan on Paperclip-generated code this week. Revisit in next cycle.

## Deep Dive 2: MCP STDIO Vulnerability — Architectural Flaw

### What It Is
OX Security's comprehensive report reveals the MCP STDIO vulnerability is not a bug but an architectural design decision in Anthropic's official SDK. The flaw enables RCE on any system running a vulnerable MCP implementation. Scale: 150M+ downloads, 200K+ vulnerable instances, 10 CVEs issued.

Four attack vector families:
1. **Unauthenticated UI injection** — through web interfaces of AI frameworks
2. **Hardening bypasses** — circumventing protections in tools like Flowise
3. **Zero-click prompt injection** — in IDEs (Windsurf, Cursor) where user prompts directly influence MCP JSON config
4. **Malicious marketplace distribution** — 9 of 11 MCP registries were successfully poisoned

Anthropic's response: "expected behavior." No protocol-level fix.

### How It Applies to Us
Our exposure assessment:
- **Kapture MCP**: Running on port 61822, exposed on all interfaces (per Apr 21 security audit). If Kapture uses STDIO transport, it's potentially vulnerable.
- **ClawHub skills**: Any ClawHub skill that uses MCP servers could be a vector. We need to audit installed skills.
- **OpenClaw itself**: OpenClaw's tool system is not MCP-based internally, but some skills may integrate MCP servers.
- **Cursor/Windsurf**: If Alex or anyone on the team uses these IDEs, the zero-click vector is relevant.

### Implementation Path
1. Audit Kapture MCP configuration — check transport type (STDIO vs HTTP). Difficulty: trivial.
2. Bind Kapture to localhost only (it's currently on all interfaces). Difficulty: trivial.
3. Audit ClawHub installed skills for MCP usage. Difficulty: moderate.
4. Add MCP security check to the healthcheck skill. Difficulty: moderate.

### Trade-offs
- Binding Kapture to localhost may break remote access if Alex uses it from his phone
- Disabling MCP entirely would lose integration capabilities
- The vulnerability is most dangerous for public-facing services — our services are behind a firewall

### Recommendation
**Do it now (partial).** Bind Kapture to localhost immediately (trivial, matches the Apr 21 security audit recommendation). Audit ClawHub skills this week. Full MCP security integration into healthcheck skill can be queued.
