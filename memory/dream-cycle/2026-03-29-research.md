## Deep Dive: Claude Code `git reset --hard` Bug

### What It Is
Claude Code v2.1.87 silently performs `git fetch origin && git reset --hard origin/main` every 10 minutes using programmatic git operations (libgit2, no external git binary). This destroys all uncommitted changes to tracked files. The timer fires at exact 10-minute intervals from session start. Evidence: 95+ reflog entries across 4 sessions over 36 hours. Verified by independent reproduction with fswatch monitoring. No external process is responsible — confirmed by lsof (only Claude Code PID has CWD in affected repo) and zero git processes at 0.1s polling intervals.

Workaround: git worktrees are immune (worktree reflog shows zero reset entries). Untracked files also survive.

### How It Applies to Us
We spawn Claude Code as a coding agent via OpenClaw's coding-agent skill. If the spawned Claude Code process exhibits this behavior, it could wipe uncommitted work in project directories (HoldPlease, Batman, any new project). We need to verify:
1. Does this affect our Claude Code version? (Check `claude --version`)
2. Does this trigger when Claude Code is spawned in `--print` mode (our default)?
3. Are our repos at risk? (We already commit frequently, which mitigates)

### Implementation Path
1. **Immediate:** Check our Claude Code version (`claude --version`) — trivial
2. **Immediate:** Add note to coding-agent SKILL.md: "Commit frequently. Known bug in CC v2.1.87 silently resets to origin/main every 10 min."
3. **Short-term:** Consider using git worktrees for all agent-spawned work
4. **Short-term:** Monitor GitHub issue #40710 for Anthropic's fix

**Difficulty:** Trivial (awareness + note) / Moderate (worktree adoption)

### Trade-offs
- Worktrees add complexity but provide bulletproof isolation
- Frequent commits are already good practice but add noise to git log
- Could pin Claude Code to a known-safe version until fix ships

### Recommendation
**Do it now** — add warning to coding-agent skill, check our version. **Queue** worktree adoption for when we start multi-agent parallel builds.

## Deep Dive: NousResearch hermes-agent

### What It Is
A self-improving AI agent framework by Nous Research that directly competes with OpenClaw. Key differentiators:

1. **Closed Learning Loop:** Automatically creates skills from complex tasks, improves them during use, and nudges itself to persist knowledge. This is what our dream cycle does manually, but hermes-agent does it automatically.

2. **Honcho User Modeling:** Uses [Honcho](https://github.com/plastic-labs/honcho) for dialectic user modeling — building a deepening model of who you are across sessions. This is a more sophisticated version of our USER.md approach.

3. **Six Terminal Backends:** Local, Docker, SSH, Daytona, Singularity, Modal. The serverless options (Daytona, Modal) let the agent hibernate when idle — zero cost between sessions.

4. **agentskills.io Standard:** Compatible with an open standard for agent skills (compare: OpenClaw uses ClawHub). Cross-framework skill portability.

5. **Migration Path:** Explicitly offers `hermes claw migrate` for OpenClaw users.

6. **Research-Ready:** Batch trajectory generation, Atropos RL environments. Designed for training next-gen tool-calling models.

### How It Applies to Us
Several ideas worth stealing:

- **Automatic skill creation:** After completing a complex task, hermes auto-generates a reusable skill. We could add this to our coding-agent or general workflow — when a multi-step task completes successfully, auto-draft a SKILL.md for it.

- **Periodic memory nudges:** The agent proactively persists knowledge without being told. Our active-context.md is similar but manual. We could add auto-flush triggers.

- **Dialectic user modeling (Honcho):** Instead of a flat USER.md, model the user as a dialogue that deepens over time. Relevant for Mirror's consciousness expansion vision.

- **Serverless backends:** If we ever move off Alex's Mac mini, Modal/Daytona could dramatically reduce costs. Agent sleeps until messaged, wakes in seconds.

### Implementation Path
1. **Study Honcho:** Read the Honcho repo, understand dialectic user modeling. Could inform Mirror's user model. — moderate effort
2. **Auto-skill creation:** After any multi-step task, prompt "Should I create a skill for this?" — moderate effort
3. **Memory nudge system:** Add periodic "flush important context" triggers to heartbeat — trivial
4. **Serverless evaluation:** Benchmark Modal/Daytona for our workload. Not urgent (Mac mini works fine). — significant effort, defer

**Difficulty:** Moderate overall. Cherry-pick ideas, don't migrate.

### Trade-offs
- Migrating to hermes-agent would mean leaving OpenClaw's ecosystem (50+ integrations, ClawHub, community)
- OpenClaw is more mature for our specific use case (iMessage, BlueBubbles, macOS-native)
- Hermes has better theoretical foundations (RL training, Honcho) but less production mileage
- The `hermes claw migrate` command suggests they see OC users as their target market

### Recommendation
**Watch it.** Don't migrate. Cherry-pick the best ideas: auto-skill creation, memory nudges, Honcho user modeling concept for Mirror. Keep hermes-agent on the radar as competition validation that the "personal AI agent" space is growing.

## Deep Dive: Microsoft VibeVoice (Open-Source Voice AI)

### What It Is
Microsoft's open-source voice AI family:

1. **VibeVoice-ASR:** Unified speech-to-text that handles 60-minute long-form audio in a single pass. Generates structured transcriptions with Who (speaker diarization), When (timestamps), and What (content). Supports 50+ languages. Now in HuggingFace Transformers v5.3.0. User-customizable context.

2. **VibeVoice-Realtime-0.5B:** Real-time streaming TTS. Supports 9+ languages and 11 English style voices. 0.5B parameter model — runs on consumer hardware.

3. **VibeVoice-TTS:** Long-form multi-speaker TTS up to 90 minutes with 4 distinct speakers. (Code removed due to misuse concerns, but model weights available.)

Community adoption: "Vibing" voice input app built on VibeVoice-ASR (macOS + Windows). 27K+ GitHub stars.

### How It Applies to Us

**ASR for HoldPlease:** Currently using OpenAI Whisper API for call transcription. VibeVoice-ASR offers:
- Free (no API costs)
- Speaker diarization (who said what — critical for hold calls)
- Timestamp alignment
- Custom context injection (tell it "this is a customer service call about lost luggage")
- 60-min single-pass (our Lufthansa calls can run 30+ min)

**TTS for AVERY/HeyX:** Currently locked into ElevenLabs for all voice. VibeVoice-Realtime could provide:
- Free local TTS for non-Debra voices
- Style variety (11 English voices)
- Lower latency for real-time conversations
- But: no voice cloning (can't replicate Debra's custom voice)

**Cost impact estimate:**
- Current Whisper API: ~$0.006/min transcription
- VibeVoice-ASR: $0 (local inference)
- Current ElevenLabs: ~$0.12/min conversation
- VibeVoice TTS: $0 (local) but no voice cloning
- Net savings: meaningful for transcription, not a replacement for ElevenLabs cloned voices

### Implementation Path
1. **Install VibeVoice-ASR:** `pip install transformers>=5.3.0`, load model, test on a HoldPlease recording — moderate effort, 1-2 hours
2. **Benchmark against Whisper:** Compare accuracy, speed, diarization quality on our actual call recordings — moderate effort
3. **If better:** Replace Whisper in HoldPlease pipeline, add as openai-whisper-api skill alternative — moderate effort
4. **TTS evaluation:** Test Realtime-0.5B for AVERY or other non-cloned voices — moderate effort, defer

**Difficulty:** Moderate. Mac mini M4 should handle the 0.5B model fine. ASR model size TBD.

### Trade-offs
- Local inference means no API costs but uses Mac mini CPU/RAM
- Speaker diarization is the killer feature Whisper lacks
- No voice cloning means ElevenLabs stays for Debra and AVERY
- Model size may be too large for Mac mini to run alongside everything else

### Recommendation
**Queue it.** Install and benchmark VibeVoice-ASR against Whisper on our HoldPlease recordings. If diarization quality is good, adopt for transcription. Keep ElevenLabs for voice synthesis. Don't try to replace everything at once.

## Deep Dive: 1M Token Context at Standard Pricing

### What It Is
As of March 13, 2026, Anthropic made the 1 million token context window generally available for Opus 4.6 and Sonnet 4.6 at standard per-token pricing (no surcharge). Previously, extended context was either unavailable or carried a premium.

Also shipped in March:
- **Claude Code Memory:** Auto-remembers project context, debugging patterns, coding preferences across sessions
- **Persistent Memory:** Remembers communication preferences and user context across all interactions
- **Doubled off-peak usage limits** (March 13-28, now expired)

### How It Applies to Us
The 1M context window changes what we can load into session:

**Current approach:** Carefully curate what goes into context. AGENTS.md + SOUL.md + USER.md + TOOLS.md + IDENTITY.md + active-context.md ≈ ~15-20K tokens. Add daily notes ≈ another 10-20K. Total ≈ ~40K tokens leaving ~160K for conversation on a 200K window.

**With 1M context:** We could load:
- All of the above (~40K)
- Full MEMORY.md (~8K)
- Last 7 days of daily notes (~50-70K)
- Entire SecondBrain vault index (~10K)
- Full conversation history for active group chats (~50K)
- Project-specific context for active builds (~50K)
- Still have ~770K tokens for conversation

**What this enables:**
- Less reliance on memory_search — just have the context loaded
- Better cross-session continuity without file reads at startup
- Full codebase in context for coding tasks
- Complete email threads, not summaries

**Cost implications:**
- Input tokens cost money. Loading 200K tokens per session ≈ 5x current cost
- But: fewer tool calls (memory_search, file reads) = fewer round trips
- Net: probably cost-neutral for complex sessions, more expensive for simple ones

### Implementation Path
1. **Evaluate current context loading:** Audit what we load now and what we miss — trivial
2. **Expand session startup:** Load 7 days of daily notes + full MEMORY.md in all sessions — trivial config change
3. **Project context injection:** For coding sessions, load entire relevant codebase — moderate
4. **Cost monitoring:** Track input token usage before/after expansion — moderate

**Difficulty:** Trivial to start, moderate to optimize.

### Trade-offs
- More context = higher input token costs (Opus: $15/M input tokens → 1M context = $15 per full-context session)
- More context ≠ better responses (needle-in-haystack degradation at extreme lengths)
- Current selective loading actually works well for most tasks
- Risk of loading irrelevant context that confuses rather than helps

### Recommendation
**Do it now** (incrementally). Expand daily note loading from 2 days to 7 days. Load full MEMORY.md in all sessions (not just main). Monitor costs. Don't try to fill the full 1M — be strategic about what additional context actually helps.
