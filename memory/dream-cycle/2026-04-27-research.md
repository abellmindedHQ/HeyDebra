## Deep Dive 2: SWE-bench Verified Discontinuation

### What It Is
OpenAI's formal analysis of why SWE-bench Verified no longer measures what it claims to. Two findings: (1) 59.4% of audited problems have flawed tests (narrow or wide), and (2) frontier models (GPT-5.2, Opus 4.5, Gemini 3 Flash) have been contaminated with gold patches from training data.

Key numbers:
- 35.5% of audited tasks: narrow tests (enforce specific implementation details not in problem description)
- 18.8%: wide tests (check for functionality not specified in the task)
- Models can reproduce exact human-written gold patches, proving training contamination

### How It Applies to Us
1. **Benchmark skepticism.** When evaluating Opus 4.7 vs 4.6, or DeepSeek V4, don't use benchmark scores. Test on our actual workloads.
2. **Verification design.** The narrow/wide test problem mirrors our own testing challenges. When Paperclip agents complete tickets, the "verification" needs to be task-appropriate, not arbitrary checklists.
3. **The "nerfed" perception.** The HN thread about Opus 4.7 degradation could be the same phenomenon — the model isn't worse, but the tasks or evaluation changed. Conversely, it could be real harness issues again.
4. **SWE-bench Pro** is the successor. Worth tracking for when we evaluate coding agent capabilities.

### Implementation Path
No direct implementation needed. This is an evaluation framework update:
- When comparing models, use our own task battery (dream cycle quality, GSD accuracy, code generation, BB message formatting)
- Don't weight external benchmarks heavily in upgrade decisions

**Difficulty:** trivial (mindset change, not code change)

### Trade-offs
- **Gain:** Better model evaluation decisions
- **Cost:** Takes more effort to evaluate models ourselves vs trusting benchmarks
- **Risk:** None

### Recommendation
**Queue it.** Update our model evaluation criteria doc (when we create one) to explicitly note benchmark contamination risks. For now, continue the Opus 4.7 wait-and-see approach.

## Deep Dive 3: Mem0 Agent Memory Benchmarks (LOCOMO)

### What It Is
Mem0 published comprehensive benchmarks comparing 10 memory approaches on the LOCOMO dataset. Key findings:

| Approach | Accuracy | Latency | Tokens/conversation |
|----------|----------|---------|---------------------|
| Full-context | 72.9% | 9.87s | ~26,000 |
| Mem0g (graph) | 68.4% | 1.09s | ~1,800 |
| Mem0 | 66.9% | 0.71s | ~1,800 |
| RAG | 61.0% | 0.70s | — |
| OpenAI Memory | 52.9% | — | — |

The crucial insight: full-context is 6% more accurate but 14x slower and 14x more expensive. Graph-enhanced memory (Mem0g) closes the accuracy gap to 4.5% while being 9x faster.

### How It Applies to Us
Our current memory architecture:
- **Short-term:** active-context.md (manual)
- **Long-term:** MEMORY.md + daily logs (append-only, manual curation)
- **Semantic search:** memory_search (embeddinggemma, built-in)
- **Graph:** Neo4j (down/unused for weeks)
- **Hybrid search:** QMD (BM25 + semantic over SecondBrain)

Compared to Mem0's findings:
1. We're doing roughly "RAG + manual curation" which lands around 61% accuracy in their benchmark
2. Graph-enhanced search (Mem0g pattern) would jump us to ~68% — but our Neo4j is down
3. Asymmetric embeddings (available in OC 2026.4.26) could improve our RAG baseline

### Implementation Path
1. **Near-term:** Get Neo4j running again. Our graph data exists, it's just the service that's down.
2. **Near-term:** Upgrade OC to get asymmetric embedding support for memory_search.
3. **Medium-term:** Evaluate Mem0 or Letta as a managed memory layer vs our manual approach.
4. **Long-term:** Implement memory decay (from the HN biological decay project) to auto-deprioritize stale entries.

**Difficulty:** moderate (Neo4j restart is trivial, evaluating Mem0/Letta is significant)

### Trade-offs
- **Gain:** Better memory recall, faster retrieval, reduced stale entries
- **Cost:** Operational complexity. Another service to maintain (Mem0) or more time fixing Neo4j.
- **Risk:** Over-engineering memory when the current system works "well enough" (10/10 memory verification probes passed tonight)

### Recommendation
**Watch it.** Our manual system scores well on self-verification. The 7+ day correction streak suggests our memory is working. But: restart Neo4j (it's free performance), and upgrade OC for asymmetric embeddings. Defer Mem0 evaluation until after the OC upgrade settles.

## Deep Dive 4: Claude Code Source Leak & Security

### What It Is
Claude Code's source code was accidentally leaked, revealing prompt injection vulnerabilities in the agent-mode coding tool. Anthropic patched quickly. Details are sparse (the leak itself is being suppressed), but the key concern is that prompt injection in coding agents can lead to arbitrary code execution.

### How It Applies to Us
We run Claude Code via Paperclip agents. These agents:
- Execute code on our Mac Mini
- Have access to git, file system, shell
- Run as the `debra` user
- Have been known to create zombie processes (the stuck curl pipes incident)

If Claude Code is vulnerable to prompt injection via repo contents (malicious code in README, comments, or commit messages), our Paperclip agents could be tricked into executing arbitrary commands.

### Implementation Path
1. Verify Claude Code version: `claude --version` (should be v2.1.116+ per previous DC research)
2. Check if the security patch is included in our version
3. Audit Paperclip agent workspace permissions — can we restrict shell access?
4. Consider: should Paperclip agents run in a sandbox?

**Difficulty:** moderate (version check is trivial, sandboxing is significant)

### Trade-offs
- **Gain:** Reduced attack surface for code execution via Paperclip
- **Cost:** Sandboxing would slow agent execution and complicate file access
- **Risk:** Current risk is low (our repos are private, no external contributors), but non-zero

### Recommendation
**Queue it.** Verify Claude Code version is patched. Don't sandbox yet (overhead not justified for private repos), but add it to the security review list.
