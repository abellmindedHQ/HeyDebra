# Dream Cycle #6 — Deep Research — 2026-04-02

## Deep Dive 1: Mem0 Graph-Enhanced Agent Memory (Mem0g)

### What It Is
Mem0g is a graph-enhanced memory layer that combines selective extraction with relationship-aware retrieval. Unlike full-context (dump everything into the window) or basic RAG (chunk + vector search), Mem0g builds a knowledge graph of extracted facts and uses graph traversal to find related memories. The LOCOMO benchmark shows it achieves 68.4% accuracy (vs 72.9% full-context) at 91% lower latency and 90% fewer tokens. The key insight: relationship-based retrieval ("Alex's girlfriend" → "Hannah" → "pregnant" → "secret") outperforms semantic similarity alone.

### How It Applies to Us
Our current memory stack:
- `MEMORY.md` (curated long-term) — manually maintained
- `memory/YYYY-MM-DD.md` (daily logs) — append-only
- `memory_search` (SQLite hybrid: BM25 + vector) — good but flat

What's missing: **relationship-aware retrieval**. When I search for "Hannah's pregnancy status," I get results. But when I search for "what should I NOT mention in group chats," the connection to Hannah's pregnancy requires me to already know the link. Mem0g's graph layer would surface "pregnancy is secret" → "don't mention in groups" automatically.

### Implementation Path
1. **Evaluate Mem0 Python SDK** — install, test against our memory files (trivial effort)
2. **Build extraction pipeline** — run Mem0's extraction on daily memory files to build a fact graph
3. **Compare against memory_search** — benchmark on 20 real queries from recent sessions
4. **If better:** integrate as supplementary layer alongside existing memory_search
5. **If not:** document findings, skip

**Difficulty:** moderate (2-3 days to evaluate properly)
**Risk:** adds dependency on external Python service. Our file-based system is simple and reliable.

### Trade-offs
- **Gain:** Better cross-referencing, automatic relationship discovery, catches things flat search misses
- **Cost:** New dependency, another service to maintain, extraction pipeline needs tokens
- **Alternative:** Our existing system works. Could just improve memory_search with tagged metadata instead.

### Recommendation
**Watch it.** Read the full paper (arXiv:2504.19413). Don't adopt yet. Our memory system is functional and Alex hasn't reviewed the knowledge architecture report yet. Adding another layer before that decision is premature. Revisit after knowledge architecture session.

## Deep Dive 2: Bonsai 8B — Local 1-Bit LLM

### What It Is
PrismML's Bonsai 8B: a true 1-bit LLM (all layers, no higher-precision fallbacks) with 8.2B parameters fitting in 1.15GB. Achieves 70.5 avg benchmark score (competitive with Llama 67.1, close to Mistral3 71.0). Runs at 44 tok/s on iPhone, likely faster on M4 Mac mini. Apache 2.0. Requires forked llama.cpp.

### How It Applies to Us
Current token costs are manageable but growing:
- Opus for deep work: expensive but worth it
- Sonnet for routine: good cost/quality
- Gemini for search: paid tier, no more 429s

A local 1-bit model could handle:
- **Email triage classification** (spam/action/fyi) — currently uses Sonnet
- **Message intent detection** — "is this a question for me?"
- **Quick fact extraction** — pull dates/names from text
- **Sentiment detection** — mood signals from messages
- **Corrections classification** — is this a correction or a task?

### Implementation Path
1. Install forked llama.cpp on Mac mini M4 (trivial, ~30 min)
2. Download Bonsai 8B (1.15GB download)
3. Benchmark on our actual tasks: email classification, message triage, fact extraction
4. If quality is sufficient (>90% agreement with Sonnet on same tasks): route low-stakes tasks to local model
5. Keep Sonnet/Opus for anything requiring reasoning, composition, or external communication

**Difficulty:** moderate (1-2 days to set up and benchmark)
**Risk:** 1-bit models may struggle with nuanced tasks. The benchmark scores are averages — need to test on OUR tasks specifically.

### Trade-offs
- **Gain:** Zero API cost for classification tasks, instant latency, works offline, privacy (all local)
- **Cost:** Setup time, maintenance of llama.cpp fork, model updates, Mac mini CPU/RAM allocation
- **Reality check:** Our monthly API costs aren't the bottleneck. Alex's time and attention is. This is a "nice to have" optimization, not urgent.

### Recommendation
**Queue it.** Fun project for a weekend or when Alex wants to tinker. Not a priority over knowledge architecture, financial buildout, or Mirror product work. Add to Linear backlog.

## Deep Dive 3: OpenClaw Task Brain (v2026.3.31-beta.1)

### What It Is
OpenClaw's new "task brain" provides an OS-level task control panel with:
- Fine-grained permission control for AI self-management
- Self-scheduling capabilities
- Trust boundaries between task types
- Human-in-the-loop approval chains

This addresses the exact problem our dream-cycle self-apply policy was built to solve: how does an AI agent decide what it can do autonomously vs what needs human approval?

### How It Applies to Us
Our current permission model:
- **Do freely:** read files, web search, calendar, organize, start services, git commit
- **Do proactively:** flag issues, update memory, fix infra, Things 3 tasks
- **Ask first:** outbound messages, spending money, config changes

This is documented in AGENTS.md but enforced purely by behavioral guidelines. The task brain could formalize this:
- Trivial self-apply → automatic permission
- Outbound comms → requires approval gate
- Config changes → requires explicit human approval
- Spending → blocked without human override

### Implementation Path
1. Check current OpenClaw version (`openclaw --version`)
2. Review task brain changelog for our use cases
3. If available: map our AGENTS.md permission tiers to task brain trust levels
4. Migrate self-apply policy from behavioral → enforced
5. Test with dream cycle proposals as first use case

**Difficulty:** moderate (depends on API stability, beta quality)

### Trade-offs
- **Gain:** Hard enforcement of permission tiers (no more accidental process narration in external chats if send-message requires approval gate)
- **Cost:** Beta software, might have breaking changes, migration effort
- **Key question:** Does the task brain support our specific permission model, or does it impose its own?

### Recommendation
**Do it soon.** Check version, review docs, test in isolation. This directly addresses our #1 recurring failure (process narration leaking to external chats). If the permission system can gate outbound messages, it's worth the beta risk.
