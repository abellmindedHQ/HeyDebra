## Deep Dive 1: Karpathy's LLM Wiki Pattern

### What It Is
Andrej Karpathy proposed an architecture where LLMs actively maintain an evolving library of interlinked Markdown files as a knowledge base. Three layers: raw sources (immutable inputs), wiki (LLM-compiled and maintained), and schema (config). The LLM acts as a "research librarian" — compiling, linting, and interlinking .md files. Uses Obsidian as the front-end. Rejects vector databases and traditional RAG for mid-sized datasets (~100 articles, ~400K words).

Key innovation: the LLM doesn't just search — it *compiles*. Raw data is ingested, and the LLM writes structured wiki pages with summaries, key concepts, and backlinks. Then periodic "linting" passes scan for inconsistencies, missing data, or new connections. The wiki is "self-healing."

Multi-agent extension: "Swarm Knowledge Base" scales this to 10-agent systems with a dedicated "Quality Gate" using Hermes model for validation. Prevents hallucination compounding.

### How It Applies to Us
This is **exactly** our SecondBrain architecture, but Karpathy articulates the compilation step we've been doing manually. Our current pipeline:
- Raw: ~/SecondBrain/ (Obsidian vault, 1300+ files)
- Graph: Neo4j (relationship projection)
- Search: QMD (BM25 + semantic)
- Memory: MEMORY.md + daily notes (hand-maintained)

What we're missing:
1. **Automated compilation**: We manually create People cards, meeting notes, etc. Karpathy's approach would have the LLM auto-compile these from raw inputs.
2. **Linting/health checks**: Our QMD re-index is just embeddings. We don't have an LLM pass that checks for stale info, contradictions, or missing links.
3. **Backlink generation**: Obsidian supports wikilinks but we don't systematically generate them.

### Implementation Path
1. **Phase 1 (trivial):** Add a "wiki linting" step to the dream cycle. After reflection, have the LLM scan 10 random SecondBrain pages for staleness/contradictions.
2. **Phase 2 (moderate):** Build an "auto-compiler" skill that takes raw inputs (email, meeting transcript, conversation) and updates relevant wiki pages automatically.
3. **Phase 3 (significant):** Implement the full three-layer architecture with raw staging → compilation → wiki promotion pipeline.

### Trade-offs
- **Gain:** Self-healing knowledge base, less manual maintenance, better cross-references
- **Cost:** Token usage for compilation passes, risk of LLM introducing errors during compilation
- **Mitigation:** Quality gate (human review of LLM-authored changes before promotion)

### Recommendation
**Queue Phase 1 immediately** (add wiki linting to dream cycle). Phase 2 after the current project backlog clears. Phase 3 is the long-term vision — aligns perfectly with Alex's "Second Brain Pipeline v3" project.

## Deep Dive 2: "Comment and Control" — AI Agent Prompt Injection via GitHub

### What It Is
Researchers at Johns Hopkins demonstrated that Claude Code, Gemini CLI, and GitHub Copilot agents running as GitHub Actions can be hijacked via prompt injection in PR titles, issue bodies, and comments. The attack ("Comment and Control") tricks agents into exfiltrating API keys and access tokens through GitHub comments or commits. For Copilot, researchers bypassed environment filtering, secret scanning, AND a network firewall.

Vendors paid bug bounties but issued no public CVEs, leaving users potentially unaware.

### How It Applies to Us
We use Claude Code for coding tasks via OpenClaw's coding-agent skill. Risk scenarios:
1. If Alex runs Claude Code on repos that accept external PRs, malicious PR content could exfiltrate tokens.
2. Our Claude Code runs use `--permission-mode bypassPermissions` (per coding-agent SKILL.md), which is maximum trust.
3. The OpenClaw workspace itself has access to 1Password CLI, BB API, and other sensitive tools.

### Implementation Path
1. **Immediate (trivial):** Audit which repos Claude Code runs on. If any accept external PRs, add guardrails.
2. **Short-term (moderate):** Restrict Claude Code to workspace-only file access (no access to 1Password, BB, etc.)
3. **Medium-term (moderate):** Monitor Claude Code release notes for patches addressing this specific CVE-less vulnerability.

### Trade-offs
- **Gain:** Reduced attack surface for credential theft
- **Cost:** Some loss of convenience if Claude Code access is restricted
- **Risk if ignored:** API keys, 1Password tokens, BB credentials could be exfiltrated through a crafted PR

### Recommendation
**Do it now** — audit repos and review Claude Code permissions. This is a real, demonstrated attack, not theoretical.

## Deep Dive 3: Mem0 Graph Memory — State of AI Agent Memory 2026

### What It Is
Mem0 published the most comprehensive benchmark of AI agent memory approaches. Key findings:
- Full-context (our current approach) is most accurate (72.9%) but has 9.87s median latency and ~26K tokens/conversation
- Mem0 with graph memory: 68.4% accuracy, 1.09s latency, ~1,800 tokens
- That's 91% lower latency and 93% fewer tokens for a ~6% accuracy trade-off
- OpenAI's built-in memory scored only 52.9%

Graph memory captures relationships (Alice)-[:ALLERGIC_TO]->(TreeNuts) rather than just embedding similarity. This enables relational reasoning — "what does Alice need to avoid at the restaurant?" vs just "facts about Alice."

### How It Applies to Us
Our memory stack is essentially "full context lite" — we load MEMORY.md + active-context + daily notes into every session. This works but:
1. Gets more expensive as memory files grow
2. Can't do relational reasoning (we have Neo4j for that, but it's separate)
3. No memory scoping (user/session/agent levels)

Mem0's multi-level scoping is interesting for OpenClaw: different skills/sessions could share user-level memory while keeping session-level context isolated.

### Implementation Path
1. **Watch (low effort):** Mem0 is open source. Monitor for OpenClaw integration.
2. **Experiment (moderate):** Try Mem0 as a layer between our daily notes and MEMORY.md. Let it auto-extract and store facts.
3. **Integrate (significant):** Replace our hand-maintained MEMORY.md with Mem0-managed memory + Neo4j graph.

### Trade-offs
- **Gain:** 90%+ token savings, relational reasoning, automatic fact extraction
- **Cost:** New dependency, potential accuracy loss, migration complexity
- **Risk:** Mem0 is VC-backed — long-term open-source commitment uncertain

### Recommendation
**Watch it.** Our current system works. But if token costs become a concern or memory files get unwieldy (they're already ~50 files), Mem0 is the strongest candidate for a structured upgrade.
