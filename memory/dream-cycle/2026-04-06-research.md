## Deep Dive #1: Claude Agent SDK with PreToolUse Hook Pattern

### What It Is

Claude Agent SDK now includes **PreToolUse hooks**, message-level interceptors that fire before any tool call executes. These hooks allow agents to:
1. Inspect the tool call before execution
2. Validate parameters (security, format, constraints)
3. Conditionally block or modify tool calls
4. Log/audit sensitive operations
5. Implement hard gates on message routing

This is different from traditional post-execution logging. PreToolUse runs BEFORE the tool fires, enabling preventive control.

### How It Applies to Us

Debra's recurring failure pattern: **process narration leaks into external chats** (3+ incidents since Apr 1). Root cause is behavioral (forgetting the rule), but external guardrails (preflight.sh, AGENTS.md rules) don't fully prevent the failure.

**PreToolUse hook solution:**
- Implement a hook that intercepts all `message.send` tool calls
- Check: destination channel is external? If yes, does the message contain internal process markers (debugging, troubleshooting, narration)?
- If detected, BLOCK the send and instead route to Alex's session with a note: "Trying to send process narration externally. Did you mean to compose this differently?"
- This is a **hard gate**, not a reminder. It prevents the error at the point of execution.

**Implementation path:**
1. Define "process narration" patterns: "Let me try...", "I'm attempting...", "Error was...", multi-message sequences with incremental debugging
2. Add PreToolUse hook to message tool calls in gateway config
3. Test against historical incidents (Apr 1 messages, Hannah thread, Marshall texts)
4. Fine-tune false positive rate (some debug messages ARE intentional)

### Implementation Path
- **Effort:** moderate (hook infrastructure is in place, needs pattern tuning)
- **Risk:** false positives if pattern matching too broad; could block legitimate messages
- **Cost:** ~0 (uses existing Claude Agent SDK feature, no additional API calls)
- **Timeline:** 1-2 hours integration + testing

### Trade-offs
- **Gain:** Hard prevention of process narration leaks. Behavior change becomes impossible, not just encouraged.
- **Cost:** Requires tuning to avoid false positives. Alex might see extra alerts initially.
- **Complexity:** Adds a new intercept point in gateway config.

### Recommendation
**Do it now.** This is the #1 recurring failure pattern (3x in 6 days). PreToolUse hook is the right architectural solution. It's low-cost and high-impact. Should be implemented before the next similar incident.

## Deep Dive #2: Neo4j Aura Agent (Graph-Grounded Agent Construction)

### What It Is

Neo4j Aura Agent is a **managed agent-building platform** that integrates directly with AuraDB (Neo4j's hosted database). Key capabilities:

1. **Automated agent construction from ontologies:** Examine your knowledge graph schema, auto-generate agent prompts, tool definitions, and query templates
2. **Semantic grounding:** Agents reason over graph relationships, not just embeddings. Reduces hallucinations.
3. **Tight agent-data integration:** Single-click MCP Server deployment for secure, hosted agent endpoints
4. **Parameterized query templates:** Pre-defined Cypher query patterns that agents can invoke safely
5. **Text-to-graph-query:** Agents can translate natural language to Cypher automatically
6. **Built-in testing environment:** Integrated testing framework against live data

**Availability:** GA as of Feb 2026, free tier available.

### How It Applies to Us

Our current architecture: SecondBrain Obsidian vault + Neo4j graph + QMD semantic search + memory_search CLI.

**Aura Agent opportunity:**
- Replace our custom Cypher queries and graph traversal logic with auto-generated, tested agent queries
- Simplify memory search: instead of semantic embedding → manual graph filtering, use "reason over graph with language understanding"
- Enable secondary agents (not just Debra) to query SecondBrain safely. (Future: HeyAvery, HeyMerle agents all use the same SecondBrain graph)
- Auto-generate Debra agent tools from ontology (People, Projects, Meetings, Concepts) → no more manual tool definitions

**Specific use cases:**
1. **Relationship reasoning:** "Who is connected to X through Y relationship?" — Aura Agent does this better than embedding search
2. **Temporal reasoning:** "What happened with X in the last 90 days?" — Graph queries excel here
3. **Multi-hop reasoning:** "Who does X know who also works on Y?" — Aura is built for this
4. **Agent-to-agent handoff:** Multiple agents querying same SecondBrain graph safely

### Implementation Path
- **Prerequisite:** Our Neo4j graph needs to be in AuraDB (currently on localhost:7474)
- **Step 1 (2 hours):** Export SecondBrain graph from local Neo4j → AuraDB
- **Step 2 (1 hour):** Define ontology in Aura Agent UI (map People/Projects/Concepts/Meetings nodes)
- **Step 3 (1 hour):** Test auto-generated agent queries against sample prompts
- **Step 4 (2 hours):** Integrate Aura Agent MCP Server into Debra's agent config
- **Effort:** 6 hours total for proof-of-concept
- **Difficulty:** moderate (Neo4j export is the main blocker)

### Trade-offs
- **Gain:** Better semantic + graph reasoning, multi-agent capability, auto-generated tool definitions, managed hosting
- **Cost:** Migration from local Neo4j to AuraDB (upfront effort). Monthly AuraDB cost if not within free tier. Loss of local-only privacy (though acceptable for SecondBrain).
- **Risk:** Aura Agent is still relatively new; pricing/feature changes possible.

### Recommendation
**Queue it.** This is a solid 2-3 week project. Implement PreToolUse hooks first (this cycle), then migrate Neo4j to AuraDB and Aura Agent integration (next cycle). The upside is cleaner multi-agent architecture and better semantic reasoning, but it's not blocking.

## Deep Dive #3: Claude Opus 4.6 Extended Thinking Mode for Dream Cycles

### What It Is

Claude Opus 4.6 and Sonnet 4.6 now feature **extended thinking mode** (also called "visible thinking"), allowing the model to:
1. Show its reasoning process step-by-step
2. Use a configurable "thinking budget" (tokens allocated to reasoning before responding)
3. Self-correct intermediate reasoning
4. Handle complex problems with more transparency

Example: asking for a strategic recommendation, the model spends 5,000 tokens thinking through trade-offs, then responds with explicit reasoning. Available via API.

### How It Applies to Us

Dream cycle Phase 4 (proposals synthesis) is currently a single-pass process: scan findings → reflect on performance → make proposals. Opus 4.6 extended thinking could:

1. **Improve proposal quality:** Let the model think deeply about conflicts, trade-offs, and interactions between proposals before finalizing. Current approach sometimes generates redundant or contradictory proposals.

2. **Reduce proposal backlog:** With better thinking, fewer half-baked proposals get queued. More concise, higher-confidence recommendations.

3. **Cost optimization:** Switch Phase 1-2 (scan/reflect) to Sonnet 4.6 with thinking, Phase 4 (synthesis) stays Opus. Current config uses Opus for all phases.

4. **Transparency:** Users (Alex) can see the reasoning behind proposals, not just the recommendation. Builds trust.

**Example improvement:**
- Current: "Proposal #5: Implement PreToolUse hooks" (1 sentence)
- With thinking: "Considered MCP integration vs PreToolUse hooks vs behavioral rules. PreToolUse is best because: [explicit reasoning]. Tradeoff: false positives can be tuned with [approach]. Estimate: [time]. Confidence: [%]."

### Implementation Path
- **Step 1 (1 hour):** Add `thinking` parameter to Opus 4.6 call in Phase 4 logic (gateway cron payload)
- **Step 2 (1 hour):** Set thinking budget (suggest 10,000 tokens for phase 4 synthesis)
- **Step 3 (2 hours):** Review first 3 cycles with thinking enabled, tune budget and prompts
- **Cost:** ~$0.03 per cycle additional (thinking tokens are 20% cheaper than regular output tokens for Opus)
- **Effort:** 4 hours total

### Trade-offs
- **Gain:** Better proposal quality, reduced backlog, lower cost, transparency
- **Cost:** Slightly longer cycle time (model thinks first). ~2 min overhead per cycle.
- **Simplicity:** Easy to enable/disable based on need

### Recommendation
**Do it next cycle.** This is a low-risk, high-upside improvement to dream cycle quality. Won't block current proposals (can enable retroactively), and cost is negligible. Aim for cycle #9 or #10.
