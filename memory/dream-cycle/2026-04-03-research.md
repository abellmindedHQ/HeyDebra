## Deep Dive 1: Claude Agent SDK

### What It Is
Anthropic took Claude Code's internal architecture and shipped it as a Python + TypeScript library. Not a wrapper around the API — the actual agent loop, tool execution, context management, and error recovery from Claude Code, made programmable.

Key architectural decisions:
- **9 built-in tools** (Read, Write, Edit, Bash, Glob, Grep, WebSearch, WebFetch, AskUser) — zero setup needed
- **18 lifecycle hooks** (PreToolUse, PostToolUse, Stop, SubagentStart, SubagentStop, Notification, PreCompact, etc.) with regex matchers and async support
- **Custom tools as in-process MCP servers** — no separate process, no network overhead, MCP protocol internally
- **Two entry points:** `query()` for fire-and-forget, `ClaudeSDKClient` for production (bidirectional, hooks, lifecycle control)
- **Subagent spawning** with isolated context windows, results flow back to parent
- **Session persistence** via `session_id` — resume conversations with full context

The hook system is the standout. A `PreToolUse` hook with a "Write|Edit" matcher can block .env modifications. `hookSpecificOutput` controls the operation (allow/deny/modify), while top-level `systemMessage` tells Claude WHY so it can adapt. Deny always wins across chained hooks. Async hooks fire side effects without blocking.

### How It Applies to Us
1. **Process narration prevention (HIGH):** A PreToolUse hook on our message-sending tool could enforce the "no narration in external chats" rule as a hard gate, not a behavioral suggestion. This is exactly the structural fix I've been unable to achieve through rules alone.
2. **Subagent architecture:** Mirrors our sessions_spawn pattern. Could use Agent SDK subagents for dream cycle phases instead of OpenClaw sessions_spawn for tighter control.
3. **Session persistence:** Similar to our active-context.md approach but built into the SDK. Could simplify our cross-session continuity.
4. **Custom tools as MCP servers:** Could wrap our BB messaging, Things 3, and gog tools as Agent SDK tools for a unified agent experience.

### Implementation Path
1. Install Claude Agent SDK (Python or TypeScript) on Mac mini
2. Build a proof-of-concept: PreToolUse hook that gates all message tool calls
3. Test against the process-narration pattern (simulate a coding task in a group chat)
4. If successful, propose to Alex as the structural fix for our #1 failure mode
5. Longer term: evaluate whether Agent SDK should complement or replace parts of our OpenClaw workflow

**Difficulty:** moderate (SDK is well-documented, but integrating with OpenClaw's existing tool chain requires careful planning)

### Trade-offs
- **Gain:** Hard enforcement of message discipline via hooks. Production-grade subagent orchestration. Session persistence built-in.
- **Cost:** Another runtime to maintain. Potential overlap/conflict with OpenClaw's native tooling. Learning curve for hook architecture.
- **Key question:** Does OpenClaw already support or plan to support PreToolUse-style hooks? If yes, we should use native hooks rather than adding another layer.

### Recommendation
**Queue it.** Research more when Alex is back. The hook system is the killer feature for our #1 problem. But adding another agent runtime needs Alex's architectural blessing.

## Deep Dive 2: Google TurboQuant

### What It Is
A two-stage, training-free, model-agnostic compression pipeline from Google Research (ICLR 2026):

**Stage 1 - PolarQuant:** Converts vectors from Cartesian to polar coordinates ("Go 5 blocks at 37°" instead of "Go 3 East, 4 North"). This exploits the fact that angle distributions are concentrated and predictable, eliminating the normalization overhead that traditional quantizers require. Uses most of the bit budget here.

**Stage 2 - QJL (Quantized Johnson-Lindenstrauss):** Uses just 1 bit per residual to eliminate bias from Stage 1's error. Zero memory overhead. Acts as a mathematical error-checker.

Results: 6x KV cache compression, 8x speed improvement, zero accuracy loss. Tested on Gemma and Mistral across LongBench, RULER, ZeroSCROLLS, Needle in a Haystack, L-Eval.

Also works for vector search indices — compresses high-dimensional embeddings with state-of-the-art recall.

### How It Applies to Us
1. **QMD vector search (HIGH):** Our QMD index is 46MB with 9,634 vectors. TurboQuant could compress this by ~6x (to ~8MB) while maintaining recall quality. Faster semantic searches across SecondBrain.
2. **Neo4j vector indices:** If we're storing embeddings in Neo4j (which the knowledge architecture report suggests), TurboQuant could keep the graph database lean.
3. **Local LLM inference:** Combined with Bonsai 1-bit LLM from last cycle, TurboQuant on the KV cache could make local models genuinely viable on the Mac mini.
4. **Cost reduction:** If we move to self-hosted embeddings (vs Gemini embedding API), compressed indices mean less RAM, less compute.

### Implementation Path
1. Install `turboquant` Python package (pip install turboquant)
2. Test with QMD's existing embeddings — can we compress the index without rebuilding?
3. Benchmark: search quality (recall@10) before and after compression
4. If successful, update QMD config to use compressed indices by default
5. Explore KV cache compression for any local model experiments

**Difficulty:** trivial to moderate (drop-in Python package, but QMD integration depends on how the index is stored internally)

### Trade-offs
- **Gain:** 6x smaller vector indices, faster semantic search, enables local LLM viability
- **Cost:** New dependency, need to verify QMD compatibility, potential edge cases with our specific embedding model (embeddinggemma-300M)
- **Key question:** Does QMD's SQLite index format support pluggable compression, or would we need to modify the indexer?

### Recommendation
**Do it soon.** Low risk, high reward. The Python package is available now. Test with QMD as first target. Could be a quick win for search performance.

## Deep Dive 3: Home Assistant 2026.4

### What It Is
Major HA release (Apr 1, 2026) with several features relevant to our setup:

1. **Native IR support:** ESPHome-powered IR proxies can control any IR device (TVs, ACs, fans, projectors). Device shows up as a real entity in HA with full state tracking.
2. **AI thinking transparency:** Assist now shows the AI's reasoning process — tool calls, results, step-by-step logic in a "Show Details" popup.
3. **Purpose-specific automation triggers/conditions:** Cross-domain triggers that think in "real-world concepts" not technical domains (almost feature complete via HA Labs).
4. **Matter lock management:** Full PIN code management for Matter-compatible locks.
5. **Dashboard improvements:** Background colors for sections, favorites on cards, gauge redesign, auto-height cards.

### How It Applies to Us
1. **IR control for Nebula 4K projector (HIGH):** The projector has an IR remote but no smart integration yet. An ESPHome IR proxy (~$5-10 for an ESP32 + IR LED) could make it fully controllable via HA and therefore via OpenClaw. Movie nights automated.
2. **IR control for LG OLED:** Same approach. The OLED may already have network control via LG WebOS integration, but IR adds reliability as a backup.
3. **AI thinking for debugging:** When we integrate OpenClaw with HA more deeply, seeing the AI's reasoning for automation decisions would help troubleshoot smart home issues.
4. **Automation triggers:** Cross-domain triggers could simplify "when Alex arrives home AND it's after sunset, turn on living room lights" type automations.

### Implementation Path
1. Update HA to 2026.4 (check current version first)
2. Order ESPHome ESP32 board + IR LED/receiver module (~$10 on Amazon)
3. Flash with ESPHome, learn IR codes from Nebula remote
4. Add projector as HA entity, build automations (movie mode, bedtime, etc.)
5. Expose to OpenClaw via HA API for voice/chat control

**Difficulty:** moderate (hardware + firmware + HA config, but well-documented path)

### Trade-offs
- **Gain:** Control previously unreachable devices, better automation triggers, AI debugging transparency
- **Cost:** Hardware purchase ($10-15), setup time (~2-3 hours), ESPHome learning curve
