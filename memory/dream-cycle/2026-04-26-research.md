## Deep Dive: Claude Opus 4.7 — Self-Verification & xhigh Reasoning

### What It Is
Opus 4.7 is Anthropic's latest production model, GA since Apr 16. The standout capabilities:
- **Self-verification:** The model devises ways to verify its own outputs before reporting back. This is native, not prompt-engineered.
- **xhigh reasoning effort:** New effort level above "high" for finer control over reasoning depth and latency.
- **Task budgets (public beta):** Set compute limits per task on the Claude Platform API.
- **13% coding resolution lift** over Opus 4.6 on 93-task benchmark. Solved 4 tasks neither Opus 4.6 nor Sonnet 4.6 could.
- **Better vision:** Higher-resolution image understanding.
- **Stronger async workflows:** Designed for long-running, multi-step tasks with rigor and consistency.

Key testimonials: "Low-effort Opus 4.7 ≈ medium-effort Opus 4.6" (Hex). Financial platforms report "game-changing speed and precision." 

Pricing unchanged: $5/$25 per M input/output tokens.

### How It Applies to Us
This is the most relevant finding for our stack:

1. **Self-verification → fixes our #1 correction pattern.** "Report-without-verifying" has 6 occurrences in corrections.md. Opus 4.7's native self-verification means the model would naturally check its own work before reporting to Alex. This isn't a band-aid; it's a model-level fix.

2. **xhigh reasoning → better multi-step task execution.** Debra's daily work (email triage → action item extraction → Things 3 update → memory write) involves chained tool calls. Higher reasoning depth reduces cascading errors.

3. **Better vision → Paperclip visual QA.** Brand identity work (ABE-35, ABE-32) involves evaluating visual assets. Opus 4.7's improved vision could let agents actually verify visual output quality.

4. **Async workflow improvements → Paperclip reliability.** Agents running long coding tasks (homepage redesign, platform MVP) benefit from the consistency and rigor improvements.

5. **Task budgets → cost control.** Could set per-task token limits on Paperclip agents to prevent runaway token consumption.

### Implementation Path
1. **Sandboxed test (moderate effort):** Run Debra on Opus 4.7 in an isolated session with representative tasks: message drafting, research, memory ops, coding review. Compare quality, tokens, latency.
2. **Paperclip agent test (moderate):** Switch one agent (e.g., Wren/QA) to Opus 4.7 and compare output quality.
3. **Full rollout (trivial if tests pass):** Change `anthropic/claude-opus-4-6` to `anthropic/claude-opus-4-7` in gateway config.

**Difficulty:** moderate (testing), trivial (deployment)

### Trade-offs
- **Gain:** Self-verification, better reasoning, improved vision, same cost
- **Cost:** Testing time (~2 hours), potential unknown behavioral differences
- **Risk:** Opus 4.7 has intentionally constrained cybersecurity capabilities (Project Glasswing). Unlikely to affect our use case, but worth noting.
- **Mitigation:** Sandboxed test before full rollout. Keep 4.6 as fallback.

## Deep Dive: Persistent Memory Architecture — Four-Layer Framework

### What It Is
MindStudio published a detailed guide on building persistent memory for Claude Code agents (applicable to any agent framework). The architecture separates memory into four layers:

1. **Agent Instructions** (CLAUDE.md / AGENTS.md equivalent): Identity, rules, behavior. Rarely changes. Loaded first every session.
2. **Brand Context** (shared knowledge): Voice, audience, product info. All agents read from this. Moderate change frequency.
3. **Agent Context** (specialized knowledge): Per-agent expertise. Each agent has its own file. Updates as the agent learns.
4. **Project Memory** (learned from runs): What happened, what worked, what failed. High change frequency. Append-only.

Key insight: **Mixing layers is the #1 mistake.** When agent instructions, brand context, and project memory all live in one file, you get bloat, staleness, and context confusion.

### How It Applies to Us
Our current architecture maps roughly:
- Layer 1 (Instructions): `AGENTS.md` + `SOUL.md` → ✅ Well-separated
- Layer 2 (Brand/Human Context): `USER.md` + `TOOLS.md` → ✅ Good
- Layer 3 (Agent Context): `IDENTITY.md` → ⚠️ Thin. Debra doesn't have deep specialized context.
- Layer 4 (Project Memory): `memory/` directory + `MEMORY.md` → ✅ Strong

Where we're weak:
- **Paperclip agents have NO persistent memory.** Each agent run starts fresh. They read from the workspace but don't write lessons, preferences, or project history back. Sable (CDO) keeps producing off-brand work partly because she has no memory of previous feedback rounds.
- **MEMORY.md is a mixed bag.** It contains critical lessons, infrastructure, people, and projects all in one file. It's approaching 200 lines. The four-layer model suggests splitting into: lessons.md (what I've learned), infrastructure.md (what runs where), people.md (who matters), projects.md (what's active).
- **No provenance tracking.** When I write to memory, there's no tag indicating source (Alex said it, I inferred it, web search found it, cron captured it). This matters for trust and staleness detection.

### Implementation Path
1. **Paperclip agent memory (significant):** Create per-agent context files in the platform workspace. Each agent reads its file at session start and appends lessons at session end. Requires Paperclip config changes.
2. **MEMORY.md split (moderate):** Split into focused files. Keep MEMORY.md as a slim index with cross-references.
3. **Provenance tags (moderate):** Add `[source: alex|web|inference|cron]` and `[date: YYYY-MM-DD]` tags to memory entries.

**Difficulty:** significant (full implementation), moderate (incremental)

### Trade-offs
- **Gain:** Better agent consistency, less repeated work, faster context loading, trustworthy memory
- **Cost:** Migration effort, larger file count, more maintenance
- **Risk:** Over-engineering. Our current system works. Splitting too aggressively could make things harder to find.

### Recommendation
**Queue it — incremental approach.** Start with provenance tags on new memory entries (trivial). Then evaluate Paperclip agent memory after the current platform work stabilizes. MEMORY.md split can wait until it hits 300+ lines.

## Deep Dive: AI Agent Memory as Engineering Discipline (Mem0 Report)

### What It Is
Mem0's "State of AI Agent Memory 2026" is the first comprehensive benchmark comparing 10 memory approaches head-to-head:
- Uses LOCOMO benchmark (standardized multi-session conversational memory evaluation)
- Measures 5 dimensions: BLEU, F1, LLM correctness, token consumption, latency
- Tests: LoCoMo baseline, ReadAgent, MemoryBank, MemGPT, A-Mem, LangMem, RAG (multiple configs), full-context, OpenAI Memory, Zep

Key findings:
- **Full-context ceiling:** Passing entire conversation history gives best accuracy but worst token consumption (26K+ tokens per query). Not production-viable.
- **Graph memory production-ready:** Relationship-based retrieval (not just semantic similarity) now competitive with full-context on accuracy while using 10-50x fewer tokens.
- **Procedural memory emerging:** How-to knowledge (not just what/when facts) as a distinct memory type.
- **Provenance tracking critical:** Multi-agent systems need to know WHO said WHAT and WHETHER it was user input or agent inference.

### How It Applies to Us
Debra's memory system is custom-built but maps to standard patterns:
- **Episodic memory:** `memory/YYYY-MM-DD.md` (daily logs of what happened) → strong
- **Semantic memory:** `MEMORY.md` (facts, people, lessons) → strong
- **Procedural memory:** `AGENTS.md` + `SKILL.md` files (how to do things) → good
- **Working memory:** `active-context.md` (current state) → unique, strong

What we're missing:
- **Graph retrieval:** Neo4j is installed but underused. QMD does hybrid BM25+semantic, but not graph-based relationship queries. "Who does Alex know who works at UTK?" requires graph, not text search.
- **Provenance:** No source tagging on memory entries.
- **Benchmark comparison:** We've never measured our memory system's recall accuracy. Could run LOCOMO-style probes (Phase 2.5 memory verification is a lightweight version of this).

### Implementation Path
1. **Get Neo4j running again (moderate):** It's been down/neglected. Restart, verify SecondBrain projection is current.
2. **Add graph retrieval to search hierarchy (significant):** QMD → Neo4j → memory_search. Use graph for relationship queries.
3. **Provenance tags (trivial):** Start tagging new entries.
4. **Self-benchmark (moderate):** Generate 50 QA pairs from last 30 days, test recall accuracy.

**Difficulty:** moderate to significant

### Trade-offs
- **Gain:** Better recall, relationship-aware search, measurable quality
- **Cost:** Neo4j maintenance, more complex search hierarchy
- **Risk:** Over-engineering for a single-user assistant. Debra serves one person. Enterprise-scale memory architecture may be overkill.

### Recommendation
**Watch and iterate.** Get Neo4j running (it's infrastructure debt). Add provenance tags (cheap). Skip the formal benchmarking for now — Phase 2.5 memory verification is sufficient for our scale. Revisit graph retrieval when the "who knows who" type queries become more frequent.
