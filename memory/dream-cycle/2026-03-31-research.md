## Deep Dive 1: Claude Code Source Leak — Architecture Insights

### What It Is
On March 31, 2026, Anthropic accidentally shipped a source map file in Claude Code npm v2.1.88, exposing 512K lines of TypeScript across ~1,900 files. The leak was discovered by security researcher Chaofan Shou. Cause: Bun runtime generates source maps by default; someone forgot to add *.map to .npmignore. This is Anthropic's second IP leak in a week (Claude Mythos exposed March 26 via CMS error).

### Key Architecture Findings (from detailed analysis)

**Agent Architecture:**
- `QueryEngine.ts`: ~46,000 lines. Handles all LLM API calls, streaming, caching, multi-turn orchestration.
- `Tool.ts`: ~29,000 lines. Defines ~40 discrete tool types with permission schemas.
- `commands.ts`: ~85 slash commands registered.

**Multi-Agent System — Three Models:**
1. **Fork model**: Byte-identical copy of parent context. Hits prompt cache. Running multiple in parallel costs ~same as one sequential (prompt cache reuse). This is brilliant.
2. **Teammate model**: File-based mailbox communication across terminal panes.
3. **Worktree model**: Each agent gets its own isolated git branch.

**Hook System:**
- 25+ lifecycle events (PreToolUse, PostToolUse, UserPromptSubmit, SessionStart, SessionEnd...)
- 5 hook types: shell commands, LLM-injected context, agent verification loops, HTTP webhooks, JavaScript functions.
- This is essentially an extension API that Anthropic hasn't publicly advertised.

**CLAUDE.md Handling:**
- Re-reads the file on every query iteration, not just at session start.
- Hierarchy: global (~/.claude/CLAUDE.md) → project-level → modular rules (.claude/rules/*.md) → gitignored local notes.
- 40,000 character limit.

**"Undercover Mode":**
- System designed to prevent leaking internal codenames (Capybara, Tengu, etc.) in public commits.
- Ironic: protected against agent leaks but not build artifact leaks.

**Bonus: "Buddy" System:**
- Complete Tamagotchi companion with species rarity, shiny variants, procedurally generated stats, soul descriptions. Behind a compile-time flag. Wild.

### How It Applies to Us
1. **Fork model for sub-agents:** OpenClaw's sessions_spawn doesn't explicitly leverage prompt cache reuse. If we could ensure parent context is shared with spawned sub-agents, we'd get massive cost savings on dream-cycle and night-swimming runs.
2. **Hook system inspiration:** Our skills are powerful but lack lifecycle hooks. PreToolUse hooks could implement the "verify recipient before send" gate I keep proposing.
3. **CLAUDE.md hierarchy:** Our AGENTS.md + SOUL.md + TOOLS.md is similar but flat. A modular rules/ directory approach could let us add context-specific rules (e.g., rules/outbound-comms.md, rules/group-chat.md) without bloating the main files.
4. **Every-iteration re-read:** We don't re-read workspace files mid-session. This means changes to active-context.md during a session aren't picked up. Could be worth exploring.

### Implementation Path
- **Modular rules directory**: Create `.openclaw/rules/` with focused rule files. Trivial effort.
- **Hook system**: Would need OpenClaw core changes. Significant effort.
- **Fork model cost optimization**: Needs investigation into OC's sub-agent prompt caching behavior. Moderate effort.

### Trade-offs
- Modular rules: more files to maintain, but each is focused and small. Net positive.
- Hooks: powerful but adds complexity. Wait for OC to implement natively.
- The leak itself is a reminder: never ship debug artifacts. Our workspace is git-tracked; ensure no secrets in commits (we have this rule, good).

### Recommendation
**Queue it.** Adopt modular rules directory this week (trivial). Watch for OC to add hook system. Study fork model for sub-agent cost optimization.

## Deep Dive 2: Letta Context Repositories & Sleep-Time Compute

### What It Is
Letta (formerly MemGPT team) has formalized a tiered memory architecture for AI agents, drawing from OS memory management principles. Their latest evolution uses "Context Repositories" — git-backed files (MemFS) that agents manage using standard tools.

**Memory Tiers:**
1. **Message Buffer** — Recent conversation messages (working memory / RAM)
2. **Core Memory** — In-context blocks with labels, descriptions, values, character limits. Editable by the agent or by specialized "memory agents." Pinned to context window.
3. **Recall Memory** — Full conversation history, searchable, auto-persisted. Not in active context.
4. **Archival Memory** — Processed/indexed knowledge in external DBs (vector or graph). Explicitly stored.

**Sleep-Time Compute:**
- Asynchronous memory management agents that run during idle periods.
- Non-blocking: memory operations don't slow conversation responses.
- Proactive refinement: reorganize and improve memory blocks when idle.
- Result: higher quality memory formation + lower response latency.

**Key insight:** Memory blocks are structured with labels, descriptions, values, and character limits. This creates an abstraction layer over the context window. Agents modify their own memory blocks using function calls.

### How It Applies to Us
Our memory architecture maps surprisingly well to Letta's tiers:

| Letta Tier | Our Equivalent | Gap |
|---|---|---|
| Message Buffer | Session context | No gap |
| Core Memory | active-context.md | We lack formal block structure (labels, limits) |
| Recall Memory | memory_search + daily files | Works well, but no auto-persistence of session content |
| Archival Memory | MEMORY.md + SecondBrain | Neo4j down, so graph retrieval broken |

**Biggest gap: Sleep-Time Compute.** Our dream cycle IS essentially sleep-time compute for self-reflection. But we don't have sleep-time memory agents that autonomously refine active-context.md, consolidate daily files, or prune stale information. We do this manually during heartbeats, inconsistently.

**Second gap: Core Memory blocks.** active-context.md is unstructured. If we gave it formal sections with character limits (e.g., "Active Tasks: max 500 chars", "Pending Decisions: max 300 chars"), it would stay cleaner and more useful.

### Implementation Path
1. **Structured active-context.md** — Add formal sections with target lengths. Trivial.
2. **Sleep-time memory agent** — Cron job that reads daily files, active-context, and consolidates/prunes. Create as a skill. Moderate effort.
3. **Auto-persistence** — Write session highlights to daily file periodically during session, not just at sign-off. Would need OC hook or heartbeat-based approach. Moderate.
4. **Archival via Neo4j** — Get Neo4j running, then build GraphRAG retrieval. Significant but foundational.

### Trade-offs
- Structured context: slightly more rigid, but prevents the 50-line "sticky note" from becoming a 200-line dump.
- Sleep-time agent: another cron job consuming tokens. But memory quality improvements should compound.
- More structure = more maintenance overhead per session. Need to find the sweet spot.

### Recommendation
**Do it now (partially).** Structure active-context.md this week. Build sleep-time memory consolidation skill within 2 weeks. Neo4j is a dependency for full archival tier.

## Deep Dive 3: DeerFlow 2.0 — Orchestration & Progressive Skill Loading

### What It Is
ByteDance's open-source "SuperAgent harness." Complete rewrite from v1 (deep-research tool) to v2 (execution engine). Built on LangGraph/LangChain. MIT licensed. ~25K GitHub stars.

**Core Architecture:**
- **Lead Agent** orchestrates, decomposes complex tasks into sub-tasks, decides parallelism, spawns sub-agents, synthesizes results.
- **Sandbox**: Each task gets an isolated Docker container with filesystem, bash terminal, Python execution. Not simulated — real execution.
- **Skills**: Markdown files defining workflows, best practices, resource references. Loaded progressively (only when relevant to current task).
- **Memory**: Persistent across sessions. Async debounced updates. TIAMAT cloud backend for enterprise scale.
- **Model-agnostic**: Works with any LLM provider.

**Progressive Skill Loading** is the key insight: instead of loading all skills into context upfront (expensive), DeerFlow loads only relevant skills based on the current task. This keeps context consumption manageable for complex multi-step workflows.

### How It Applies to Us
OpenClaw already does most of what DeerFlow does:
- Sub-agent spawning (sessions_spawn) ≈ DeerFlow lead agent
- SKILL.md format ≈ DeerFlow Skills (actually, DeerFlow adopted the SKILL.md format from the broader ecosystem)
- Workspace files ≈ DeerFlow persistent memory
- Sandbox execution ≈ DeerFlow Docker containers

**Where DeerFlow is ahead:**
1. **Progressive skill loading.** We load skill descriptions in the system prompt, but the full SKILL.md content is only loaded on demand (via read). DeerFlow formalizes this with priority scores and relevance matching. We could add skill tagging/scoring for faster matching.
2. **Async memory updates.** DeerFlow's debounced queue for memory writes prevents blocking. Our memory writes are synchronous (file write during conversation). Not a huge issue at our scale but worth noting.
3. **Task decomposition formalization.** DeerFlow's lead agent explicitly breaks tasks into a dependency graph before executing. Our sub-agent spawning is more ad-hoc.

### Implementation Path
- **Skill tagging/scoring**: Add relevance tags to SKILL.md descriptions for better matching. Trivial.
- **Task decomposition**: For complex tasks (night-swimming suite, dream cycle), create explicit dependency graphs. Moderate.
- **Async memory**: Not urgent at our scale. Watch.

### Trade-offs
- DeerFlow requires Docker for sandboxing. We run on bare metal (Mac mini). Docker overhead may not be worth it for our use case.
- LangGraph/LangChain dependency = heavier stack. OpenClaw is leaner.
- MIT license means we could borrow ideas/patterns freely.

### Recommendation
**Watch it.** Borrow the progressive skill loading concept for our skill system. Don't adopt the framework itself — OpenClaw is already doing this job for us. Study their task decomposition patterns for improving our night-swimming orchestration.
