### 1. [Claude Opus 4.7 GA — Self-Verification & xhigh Reasoning](https://www.anthropic.com/news/claude-opus-4-7)
**Source:** Anthropic Blog (Apr 16) | **Relevance:** high
**Summary:** Opus 4.7 is now generally available. Key gains over 4.6: 13% resolution improvement on coding benchmarks, better vision (higher-res image understanding), self-verification of outputs before reporting, xhigh default reasoning, and stronger performance on long-running async workflows. DC#19 proposed a sandboxed test — this is now even more compelling with GA status.

### 3. [Claude Code Harness Post-Mortem: All Fixes in v2.1.116+](https://venturebeat.com/technology/mystery-solved-anthropic-reveals-changes-to-claudes-harnesses)
**Source:** Anthropic Engineering (Apr 23) | **Relevance:** high
**Summary:** Confirmed: Claude Code degradation was caused by 3 product-layer bugs (reasoning effort downgraded to medium, caching bug clearing thinking each turn, verbosity system prompt capping output). All fixed as of Apr 20 (v2.1.116). API was never affected. Token limits reset for subscribers.
**Why it matters for us:** This was highlighted in DC#18 and DC#19 but bears repeating: the Opus 4.7 "vulnerability" reports from early April were almost certainly measured during the bug window. Paperclip agents running Claude Code should be on v2.1.116+ to get proper reasoning.

### 5. [State of AI Agent Memory 2026 — Mem0 Benchmark Report](https://mem0.ai/blog/state-of-ai-agent-memory-2026)
**Source:** Mem0 Engineering (Apr 25) | **Relevance:** high
**Summary:** Comprehensive benchmark of 10 memory approaches using LOCOMO evaluation framework. Measures BLEU, F1, LLM score, token consumption, and latency. Graph memory (relationship-based retrieval) now production-ready. Procedural memory (how-to knowledge) joining episodic and semantic as a first-class type. Memory is now a distinct engineering discipline with standardized benchmarks.
**Why it matters for us:** Debra's memory system (MEMORY.md + daily logs + corrections.md + memory_search) is our custom implementation of exactly this. The four-layer framework (instructions/brand/agent/project) maps roughly to our SOUL.md/USER.md/AGENTS.md/memory/ structure. The benchmark could help us evaluate whether our approach is competitive or if we should adopt Mem0 or similar.

### 6. [Persistent Memory System for Claude Code Agents — Four-Layer Framework](https://www.mindstudio.ai/blog/persistent-memory-system-claude-code-agents)
**Source:** MindStudio Blog | **Relevance:** high
**Summary:** Detailed guide on building persistent memory for Claude Code agents using a four-layer separation: Agent Instructions (CLAUDE.md), Brand Context (shared knowledge), Agent Context (specialized), and Project Memory (learned from runs). Key insight: mixing layers is the #1 mistake. Each has different scope, owner, and update cadence.
**Why it matters for us:** This validates our architecture (SOUL.md = identity, USER.md = brand/human context, AGENTS.md = agent instructions, memory/ = project memory). But it also highlights that Paperclip agents may benefit from their own persistent memory layers — each agent should have its own context file that persists across runs, not just the shared workspace context.

### 7. [MemMachine: Open-Source Memory Layer with Neo4j](https://neo4j.com/videos/nodes-ai-2026-memmachine-agents-that-learn-memory-that-lasts/)
**Source:** NODES AI 2026 (Apr 15) | **Relevance:** medium
**Summary:** MemMachine is an open-source memory layer using Neo4j to store and refine agent memories across sessions. Enables AI apps to learn preferences, recall context, and improve over time. Solves the "start from scratch when switching models" problem.
**Why it matters for us:** We have Neo4j (localhost:7474) but it's been underutilized. MemMachine could supercharge Debra's memory by adding graph-based retrieval on top of our text-based system. Particularly interesting for people/relationship queries where connections matter more than keywords.

### 10. [Graph Memory Goes Production — HN Consensus](https://news.ycombinator.com/item?id=47912092)
**Source:** Hacker News discussion | **Relevance:** medium
**Summary:** HN consensus emerging that graph memory (retrieving facts by relationships, not just semantic similarity) is production-ready in 2026. Provenance tracking in memory layers is crucial for multi-agent setups to distinguish user input from agent inferences. "Verification as the bottleneck" continues as a dominant theme.
**Why it matters for us:** Our Neo4j SecondBrain projection is exactly this pattern — graph-based knowledge retrieval. The provenance tracking insight is relevant: when Debra writes to memory, we should track whether the info came from Alex, from a web search, from another agent, or from inference. This would help with the "stale info" problem flagged in memory verification.
