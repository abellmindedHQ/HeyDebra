## Deep Dive: NousResearch/hermes-agent

### What It Is
Hermes Agent is a self-improving AI agent framework from NousResearch, currently #1 trending on GitHub with 52,867 total stars. It's the spiritual successor to what OpenClaw is: a personal AI agent that lives across messaging platforms, runs scheduled automations, manages its own memory, and creates/improves skills from experience.

Key differentiators over OpenClaw:
- **`hermes claw migrate`** — explicit migration command for OpenClaw users. This is a competitive moat attack.
- **Honcho dialectic user modeling** — builds a deepening model of who you are across sessions. Semantic + temporal user graph, not just a flat MEMORY.md.
- **Autonomous skill creation** — after complex tasks, the agent writes a skill for what it just did, without being told to. Skills self-improve during use.
- **FTS5 session search with LLM summarization** — full-text search across ALL past conversations with an LLM-summarized result set. Better than our current semantic-only memory_search.
- **Atropos RL environments** — batch trajectory generation for fine-tuning future models. They're building training data from real usage.
- **agentskills.io compatibility** — claims to support the same open standard our skills use. Our skills might already run on Hermes.
- **Serverless persistence** — Daytona and Modal backends. Agent hibernates when idle, wakes on demand. Near-zero idle cost.
- Multi-backend deployment: local, Docker, SSH, Daytona, Singularity, Modal, $5 VPS.

### How It Applies to Us
This is a direct category threat AND a learning source.

**Threat:** 
- The `hermes claw migrate` command is a clear UX statement: "switching from OpenClaw is easy." Anyone in the OpenClaw user base who's frustrated (Claude Code rate limits, Gemini API single-point-of-failure, BB attachment bug) has a visible off-ramp.
- Hermes is model-agnostic by design (200+ models via OpenRouter). Our setup has a single-point-of-failure on Gemini that just knocked out two core tools for days.

**Learning source:**
- Their user modeling approach (Honcho) is exactly what Mirror's "Silvering" component should be. Worth studying the architecture.
- Autonomous skill creation after complex tasks = what our dream cycle tries to do manually. Could we add a post-task hook that writes a skill stub when a complex task completes?
- FTS5 + LLM summarization for session recall = better than our current qmd-only approach. We have qmd, but no LLM-summarized retrieval layer on top.
- agentskills.io compatibility → if our SKILL.md format is already close, we might be able to publish to that registry.

### Implementation Path
**Not "install Hermes" — rather, steal specific ideas:**

1. **User modeling layer** (significant effort): Build a lightweight Honcho-like system that periodically synthesizes USER.md from real session transcripts. Not just a flat file — a semantic profile with temporal decay (old facts count less).

2. **Post-task skill capture hook** (moderate effort): After any complex session, add a dream-cycle-adjacent prompt: "What skill would capture what we just did?" Write a stub SKILL.md if anything non-trivial emerged.

3. **agentskills.io exploration** (trivial): Check if our existing skills are compatible with the agentskills.io standard. If yes, publish — free discoverability.

4. **FTS5 session recall** (moderate): qmd already handles semantic search. Add a BM25-first FTS5 layer for exact phrase/name recall that currently misses.

### Trade-offs
- **Gain**: Better user modeling, skill compounding, more resilient memory
- **Cost**: Significant engineering time. These aren't trivial add-ons.
- **Risk**: Feature creep in what's already a complex system. Alex should pick 1-2 of these, not all 4.

### Recommendation
**Watch it.** Don't switch to Hermes — we have deep customization (Debra persona, ORNL context, iMessage integration via BB) that Hermes wouldn't replicate easily. But:
- Check agentskills.io compatibility NOW (trivial, 30 min)
- Steal the post-task skill capture idea for dream cycle Phase 2
- Monitor 30 days: if migration is becoming a real pattern, escalate

## Deep Dive: multica-ai/multica

### What It Is
Multica is an open-source managed agents platform — essentially a project management system where AI agents are first-class team members. Self-hostable via Docker. Key features:
- **Agent as teammate**: agents have profiles, show up on kanban boards, post comments, create issues, report blockers proactively
- **Full task lifecycle**: enqueue → claim → start → complete/fail with real-time WebSocket streaming
- **Reusable skills**: every solution becomes a team-reusable skill that compounds capability over time
- **Multi-workspace**: workspace-level isolation, own agents/issues/settings per team
- **Multi-runtime support**: explicitly names OpenClaw as a supported runtime alongside Claude Code, Codex, OpenCode

The CLI daemon (`multica daemon start`) auto-detects available agent CLIs on your PATH. You then assign issues to agents from a web UI just like you'd assign to a human on JIRA or Linear.

### How It Applies to Us
Two distinct use cases:

**1. ORNL Transformation Group pitch**
Alex leads 30 web devs at ORNL. The Transformation Group mandate is to modernize how the org builds software. Multica is literally "here's how to run a mixed human+AI dev team at scale." It integrates with OpenClaw (which Alex already runs), supports multi-workspace isolation (different teams), and provides management-friendly dashboards.

Pitch to Jay Eckles: "We can deploy Multica internally, connect it to our existing OpenClaw infrastructure, and start measuring which types of dev tasks are candidates for agent delegation. No vendor dependency — it's open-source and self-hosted."

**2. Alex's personal productivity**
For Abellminded projects (Mirror, Pools, HeyDebra), Multica could replace the ad-hoc "spawn a sub-agent in sessions_spawn" approach with a structured kanban where agents are persistent team members with skill profiles.

### Implementation Path
**For ORNL exploration (moderate effort, ~1 week):**
1. Stand up Multica locally: `docker compose -f docker-compose.selfhost.yml up -d`
2. Connect OpenClaw daemon: `multica daemon start` (auto-detects)
3. Create a small test project with 3-5 realistic ORNL-type issues
4. Run 3-5 agent sessions through it, measure quality
5. Write a 1-page brief for Jay

**For Abellminded (trivial to start, moderate to sustain):**
1. `brew tap multica-ai/tap && brew install multica`
2. Connect to Multica cloud (no Docker required for MVP)
3. Create Abellminded workspace, add Mirror + Pools projects
4. Start routing Claude Code tasks through it

**Difficulty:** moderate (Docker + setup), but well-documented

### Trade-offs
- **Gain**: Structured agent task management, skill reuse across sessions, ORNL pitch material
- **Cost**: Another service to maintain. Docker + Postgres = real infrastructure overhead.
- **Risk**: Could become orphaned tooling if Alex doesn't use it consistently.

### Recommendation
**Queue it for ORNL context.** This is a stronger play than a personal tool — the team management angle is the real value. Alex should evaluate this as part of his Transformation Group strategy, not just personal tooling. Suggest a 30-min exploration to build the pitch, then bring it to Jay.

## Deep Dive: forrestchang/andrej-karpathy-skills (CLAUDE.md Principles)

### What It Is
A single CLAUDE.md file derived from Andrej Karpathy's public analysis of LLM coding pitfalls. Trending at 1,450 stars/day (11,863 total). Four principles:

**1. Think Before Coding**
- State assumptions explicitly. Don't silently pick an interpretation.
- Present multiple interpretations when ambiguity exists.
- Push back when a simpler approach exists.
- Stop when confused — name what's unclear and ask.

**2. Simplicity First**
- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- If 200 lines could be 50, rewrite it.

**3. Surgical Changes**
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style.
- If you notice unrelated dead code, mention it — don't delete it.
- When your changes create orphans: remove only what YOUR changes made unused.

**4. Goal-Driven Execution**
- Transform imperative tasks into verifiable goals.
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- For multi-step tasks, state a plan with verify steps.
- Strong success criteria let the agent loop independently.

Installable as a Claude Code plugin:
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills
```

### How It Applies to Us

**Direct application to AGENTS.md:**
The "Think Before Coding" principle maps almost perfectly to the AGENTS.md pre-task checklist we don't have yet. Every complex task I take on should start with:
1. State my understanding of the task
2. Identify ambiguities and state assumptions
3. Propose a plan with success criteria before executing

**Direct application to coding-agent skill:**
When spawning Claude Code via the coding-agent skill, the SKILL.md should include these principles as injection text for the agent's prompt. We already have some of this, but "Goal-Driven Execution" (success criteria + verify loop) is the missing piece.

**Self-application:**
Principle 3 (Surgical Changes) is particularly relevant to how I handle file edits. I sometimes "improve" adjacent code or reformatted things that weren't in scope. This is a concrete, checkable behavior I can change immediately.

### Implementation Path
**Trivial options (can do now):**

1. Install as Claude Code plugin:
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills
```

2. Add a 4-item pre-task checklist to AGENTS.md under a new section "Pre-Task Protocol for Complex Work":
```markdown
Before starting any non-trivial task:
1. State your understanding of what's being asked
2. Identify ambiguities — ask before assuming
3. Define success criteria (what does "done" look like?)
4. Estimate scope — if it's bigger than expected, say so before diving in
```

3. Update coding-agent SKILL.md to include Goal-Driven Execution framing in sub-agent prompts.

**Difficulty:** trivial (1-2 file edits)

### Trade-offs
- **Gain**: Fewer unnecessary edits, cleaner PRs, better task handoffs to sub-agents
- **Cost**: Slightly more upfront explanation per task
- **Risk**: Over-applying rigor to simple requests. The repo explicitly notes: "these guidelines bias toward caution over speed. For trivial tasks, use judgment."

### Recommendation
**Do it now.** The plugin install is one command. The AGENTS.md pre-task checklist is 5 lines. The coding-agent skill update is 30 minutes. Return on investment is immediate and the downside is near-zero.

This is the easiest high-signal improvement in tonight's cycle.
