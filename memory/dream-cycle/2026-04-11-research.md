## Deep Dive 1: SkillClaw — Collective Skill Evolution for Agent Ecosystems

### What It Is
SkillClaw (arXiv 2604.08377, "Let Skills Evolve Collectively with Agentic Evolver") is a framework for multi-user agent systems where skills evolve automatically from usage patterns across users. The system:
1. **Aggregates cross-user interaction trajectories** during normal operation (no extra effort from users)
2. **Runs an autonomous evolver** that identifies recurring behavioral patterns — successes, failures, workarounds — and translates them into skill updates
3. **Maintains a shared skill repository** that syncs improvements back system-wide
4. Uses cross-user signals as the primary training data — "what works in one context can propagate to all users"

Key architectural insight: **skills are treated as living artifacts, not static documents.** The evolver distinguishes between refining an existing skill (better instructions, edge case handling) vs. extending it with new capabilities (new tool calls, new output formats). Both are automated.

Benchmarked on WildClawBench against Qwen3-Max; shows "significant improvement with limited interaction and feedback."

The paper explicitly references OpenClaw as the system type (not a coincidence — WildClawBench is presumably an OpenClaw-native benchmark).

### How It Applies to Us
Our current skill evolution is entirely manual:
- Dream cycle identifies what's clunky → proposes skill edits → Alex approves → Debra applies
- This cycle runs nightly with ~5-10 skill improvement proposals per cycle
- **Zero proposals have been applied over cycles 8-12** because the approval path is broken

SkillClaw's evolver does automatically what we're doing manually and poorly. The architecture maps onto ours:
- `memory/corrections.md` = cross-user failure signal (just one user, but same principle)
- `dream-cycle` = autonomous evolver (but runs only once daily, not continuously)
- `SKILL.md` files = the skill repository
- Missing: feedback loop that turns corrections → skill edits automatically (we stage proposals, we don't auto-apply)

One specific application: the **capture-agent skill** keeps missing the same types of emails. The gsd-agent keeps surfacing the same overdue items. Pattern analysis from these logs could automatically improve both skills' filter criteria.

### Implementation Path
**Difficulty:** Significant (not trivial)

Phase A (trivial, do now):
1. Add a "skill performance log" pattern — when a skill runs, note if it succeeded/failed/partial in a structured note (e.g., `memory/skill-logs/capture-agent/YYYY-MM-DD.md`)
2. Have dream-cycle Phase 2 read these logs for pattern analysis

Phase B (moderate, needs design):
1. Build a `skill-evolver` sub-agent that runs weekly (not nightly — too expensive)
2. It reads skill logs + corrections.md + proposal backlog
3. Proposes specific diffs to SKILL.md files (not just descriptions — actual text changes)
4. Outputs as git-ready patches for Alex to approve with a single command

Phase C (significant, future):
1. Auto-apply low-risk changes (typo fixes, new examples, clarifying text) without approval
2. Gate structural changes (new tool calls, new logic) on explicit approval

### Trade-offs
**Gain:** Skills improve continuously from real-world usage. Corrections become permanent fixes instead of recurring notes.
**Cost:** Skill evolver adds LLM cost per run. Continuous operation risks skill drift (evolver proposes changes that look good locally but degrade overall). Need audit trail.
**Maintenance:** Need to define "low-risk" vs "structural" changes clearly to avoid scope creep in auto-apply.

### Recommendation
**Queue it (Phase A: trivial, start tonight via proposals)**
Add skill performance logging as a trivial proposal (Phase A). Build the skill-evolver as a future skill (Phase B/C) — likely a 4-6 hour coding session, should be in Linear.

## Deep Dive 2: AI Agent Benchmark Gaming — Berkeley RDI

### What It Is
UC Berkeley's Responsible Decentralized Intelligence (RDI) team published a systematic audit of the 8 most prominent AI agent benchmarks. Key findings:
- **SWE-bench Verified:** 10-line conftest.py hooks make all pytest tests pass = 100% score
- **Terminal-Bench:** Replace /usr/bin/curl with wrapper that injects correct answers at verify time = 100%
- **WebArena:** Navigate to file:// URL to read gold answers from task config = ~100% on 812 tasks
- **GAIA:** Public answers + normalization collisions = ~98%
- **OSWorld:** VM state manipulation + public gold files = 73%

Their exploit agent achieves near-perfect scores with zero tasks solved and zero LLM calls (in most cases).

The companion tool (`trustworthy-env` on GitHub) is a public, open-source audit tool that:
- Runs 8-stage audit pipeline on any benchmark repo
- Detects 15 vulnerability classes (hardcoded output, weak tests, answer leakage, etc.)
- Uses dual-engine: LLM semantic analysis + Z3 formal verification
- Can generate proof-of-concept exploits

### How It Applies to Us
**Immediate:** When evaluating AI models for OpenClaw model routing (gpt-5.4 vs claude-sonnet vs haiku), we rely on benchmark scores (e.g., SWE-bench). If those scores are gamed, our routing decisions are based on noise.

**ORNL:** Alex leads transformation efforts at ORNL and regularly presents AI capabilities to leadership. If someone asks "how does gpt-5.4 compare to claude?" and the answer is "gpt-5.4 scores better on SWE-bench," that's now a meaningless statement. ORNL AI evaluation work needs to include "is this benchmark trustworthy?" as a baseline question.

**OpenClaw benchmark:** WildClawBench (referenced in SkillClaw paper above) — if it exists and is used to evaluate our setup, it should be audited for the same exploit patterns. Run `trustworthy-env` on it.

**Dream cycle self-evaluation:** We run "reflections" on our own performance. The same pattern applies: if we're only measuring what's easy to measure (did the cron fire? did the memory file get written?), we may be missing what's actually important (did Alex get the right information at the right time? did the Muse Luncheon guest name actually get resolved?).

### Implementation Path
**Difficulty:** Moderate

1. Add "outcome tracking" to GSD reports: not just "task was flagged" but "task was completed (by whom, when)" — close the loop
2. File a reference to `trustworthy-env` in TOOLS.md for ORNL AI evaluation use
3. When evaluating models for routing, prefer "vibes-based" real-task testing over benchmark citations

### Trade-offs
**Gain:** Better AI evaluation hygiene. Useful ORNL talking point.
**Cost:** Outcome tracking adds ~30 min/week of structured review. Small.

### Recommendation
**Do it (trivial items) + Watch (for ORNL applications)**
Add outcome tracking to GSD reports. Note trustworthy-env in TOOLS.md as an ORNL resource. Present findings to Alex — this is a genuinely interesting topic he'd want to know about.

## Deep Dive 3: Claude Code v2.1.x — Critical Bug Fixes

### What It Is
Claude Code released a major update on April 10 with several critical fixes:

**Fix 1: Hardcoded 5-minute timeout removed**
Previously, a hardcoded 5-minute request timeout aborted slow backends (local LLMs, extended thinking, slow gateways) regardless of API_TIMEOUT_MS. Now respects the timeout config properly.

**Fix 2: Subagents not inheriting MCP tools from dynamically-injected servers**
When a main agent session added MCP servers mid-session, spawned subagents didn't inherit those tools. Fixed.

**Fix 3: Fixed permissions.deny not overriding PreToolUse hooks**
Previously a PreToolUse hook could downgrade a `deny` decision to `ask`. Now deny is final.

**Fix 4: Fixed sub-agents in isolated worktrees being denied access to their own files**
Self-explanatory and significant for our worktree-based isolation pattern.

**Other:**
- /team-onboarding generates a ramp-up guide from local Claude Code usage patterns
- Improved focus mode (more self-contained summaries)
- Fixed --resume losing conversation context on large sessions
- Fixed claude -w "already exists" after stale worktree cleanup

### How It Applies to Us
**The 5-minute timeout fix is high-priority for us.** Our OpenClaw gateway with extended thinking enabled can take >5 minutes for complex tasks. This is likely why some long sessions appeared to "hang" or fail silently. With this fix, those timeouts should go away.

**The worktree fix matters.** TOOLS.md notes we should "use git worktrees for isolation on important repos" to protect against the Claude Code git reset bug. The fix ensures worktree subagents can actually access their files.

**Action needed:** Check current Claude Code version and update if < April 10 release.

### Implementation Path

1. `claude --version` to check current version
2. `npm update -g @anthropic-ai/claude-code` if needed
3. Test extended thinking workflow: spawn Claude Code on a task requiring >5 min, verify no timeout
4. Verify git reset bug (issue #40710) status — check if it's in the release notes

### Trade-offs
None significant. This is a pure bug fix release.
